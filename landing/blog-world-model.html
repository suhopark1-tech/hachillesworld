당신의 AI 에이전트,
왜 실패하는지 알고 있나요?

HAchillesWorld로 10분 안에 World Model을 진단하다

박
박성훈
2026년 6월 약 2,500자 · 10분 읽기 ◈ 실전 케이스 포함

지난달, 나는 꽤 당혹스러운 경험을 했다.

공급망 최적화를 위해 6개월 동안 공들여 만든 AI 에이전트가 발주 하나를 잘못 처리했다. 예산보다 143만 원 초과한 발주였다. 더 황당한 것은, 에이전트가 "나는 80% 확신한다"고 말하면서 그 결정을 내렸다는 사실이다.

로그를 뒤졌다. 에러 메시지는 없었다. 코드도 정상이었다. 그렇다면 무엇이 문제였을까?

진짜 문제는 코드가 아니었다

나는 에이전트가 세상을 잘못 이해하고 있었다는 것을 한참 뒤에야 알았다.

기술적으로 표현하면 이렇다: 에이전트의 World Model이 현실과 29.6% 괴리되어 있었다. 에이전트는 비용이 482만 원이 될 것이라 예측했지만, 실제로는 625만 원이었다. 그 차이가 발주 결정에 그대로 반영됐다.

더 심각한 것은, 이 괴리를 아무도 측정하지 않고 있었다는 점이다.

우리는 모델 정확도를 측정했다. 레이턴시를 측정했다. 비용을 모니터링했다. 그런데 에이전트가 세상을 얼마나 정확하게 이해하는지 — World Model의 품질은 측정하지 않았다.

이 경험이 나를 HAchillesWorld로 이끌었다.

World Model이란 무엇인가

AI 에이전트는 단순히 LLM을 호출하는 것이 아니다. 에이전트는 세상에 대한 내부 모델을 가지고 있고, 그 모델을 기반으로 행동을 계획한다.

● 에이전트의 실제 작동 방식 
현재 상태 → [World Model] → 미래 예측 → 최선의 행동 선택

World Model이 현실과 다르면:
현재 상태 → [잘못된 World Model] → 틀린 예측 → 나쁜 결정

문제는 World Model의 품질이 겉으로 드러나지 않는다는 것이다. 에러가 없다. 코드는 돌아간다. 그런데 결과가 나쁘다.

HAchillesWorld는 이 보이지 않는 문제를 수치로 만든다.

HAchillesWorld가 측정하는 것

HAchillesWorld는 에이전트를 Levels × Laws 2차원 좌표계에서 진단한다.

Levels (에이전트 지능 수준):

  • L1 Predictor: 패턴 매칭, 단순 예측
  • L2 Simulator: 멀티스텝 시뮬레이션, 계획 수립
  • L3 Evolver: 환경 변화에 자율 적응

Laws (운영 도메인):

  • Digital: API·소프트웨어·데이터 처리
  • Physical: 로봇·제조·물리 환경
  • Social: 고객·협업·다중 에이전트
  • Scientific: 연구·분석·실험

진단 결과는 이렇게 나온다:

● HAchillesWorld Scan 결과 — supply-chain-agent 
🟡 [공급망 에이전트] L2.3 × Digital Laws
   종합 점수: 67/100

   World Model 품질 : 72/100  ⚠
   에이전시 수준    : 61/100  🔴
   운영 건전성      : 68/100  ⚠

   🔴 즉시 조치 필요 (3건)
    ├─ Calibration ECE: 0.23 → 목표 0.10 미만
    ├─ Self-Correction Rate: 0% → 10% 이상 필요
    └─ Harness Coverage: 6개 → 20개 이상 필요

   💡 로드맵: L2.3 → L3 (24주, 예상 절감 ₩48M/년)

15개 지표가 자동으로 측정된다. 10분이면 충분하다.

Planning Depth: 에이전트가 얼마나 멀리 보는가

HAchillesWorld v2.0에서 새롭게 도입된 개념이 Planning Depth다.

에이전트는 몇 단계 앞까지 시뮬레이션하고 결정을 내리는가?

내 에이전트의 문제가 바로 여기 있었다. Planning Depth가 3이었다.

● EP-2847 — Depth 3의 한계 
Depth 3이 본 것:
  발주 결정 → 입고 예정 → 재고 충족 (끝)

Depth 3이 보지 못한 것:
  → 시장 가격 +29.6% 급등
  → 실제 비용 ₩6.25M (예측 ₩4.82M)
  → 예산 초과 ₩1.43M

Depth 3은 공급망 리드타임(평균 7일)이 완료되기 전에 계획이 끝난다. 결과를 보지 못하고 결정하는 구조다.

HAchillesWorld는 MCTS(Monte Carlo Tree Search)를 활용해 Planning Depth를 단계별로 끌어올린다.

Depth 공급망에서의 의미
3발주 → 입고 (리드타임 전에 끝)
8발주 → 입고 → 가격 변동 → 다음 발주 조정
15전체 조달-생산-판매 사이클 + 계절성 포함
22시장 충격 자율 감지 + 전략 자동 재계획

우리 에이전트는 24주 만에 Depth 3에서 22까지 끌어올렸다.

실제로 어떻게 작동하는가

설치는 간단하다.

bash 
pip install hachillesworld

에이전트 클래스에 데코레이터 하나를 추가한다.

instrument.py 
from hachillesworld import HAchillesWorldClient, instrument

client = HAchillesWorldClient()

@instrument(client, agent_name="my-agent")
class MyAgent:
    def plan(self, state, goal): ...
    def execute(self, action): ...
    def observe(self, result): ...
    def reflect(self, episode): ...

기존 코드 변경 없이 자동으로 모든 DEOR 루프가 기록된다. 진단 실행:

bash 
hachillesworld scan \
  --logs ./logs.json \
  --name "my-agent"

# 최적화 로드맵 자동 생성
hachillesworld optimize ./report.json

24주 동안 무슨 일이 일어났나

솔직하게 말하면, 처음 3주는 ECE(Calibration Error)를 고치는 데만 썼다.

ECE 0.23이라는 숫자가 의미하는 것: 에이전트가 "80% 확신한다"고 말할 때, 실제 정확도는 57%에 불과하다. 과신이다.

Temperature Scaling 하나로 이걸 0.09까지 낮췄다. 코드 30줄이었다.

그 다음은 하네스(Harness)였다. 에이전트가 같은 실수를 반복하지 않도록 규칙을 만들었다. 처음엔 6개였던 규칙이 Phase 1 끝에는 22개가 됐다. Phase 3에서는 Meta-Harness가 규칙을 스스로 만들었다. 8개의 규칙이 자동 생성됐고, 모두 승인했다.

에이전트 실패의 진짜 원인

대부분의 에이전트 실패는 코드 버그가 아니다. 세 가지 근본 원인이 있다.

1. World Model이 현실을 반영하지 못한다
환경이 바뀌었는데 에이전트의 내부 모델이 업데이트되지 않았다. Simulation Drift가 쌓인다.

2. 에이전트가 너무 짧게 본다
Planning Depth가 얕으면 행동의 장기적 결과를 예측하지 못한다. 단기 최적이 장기 최악이 된다.

3. 같은 실수를 반복한다
하네스(Harness)가 없으면 에이전트는 어제의 실수를 오늘도 반복한다.

HAchillesWorld는 이 세 가지를 모두 측정하고, 개선한다.

지금 시작할 수 있다

API 키 없이, 로컬에서, 무료로.

bash 
pip install hachillesworld
hachillesworld scan --logs ./my_logs.json --name "my-agent"

처음 진단 결과가 나왔을 때의 느낌을 기억한다. 막연하게 "뭔가 잘못됐다"고 느끼던 것이 수치로 보이는 순간이었다.

"Calibration ECE 0.23. 에이전트가 과신하고 있습니다."
— 이 한 줄이 다음 6개월의 방향을 바꿨다.
