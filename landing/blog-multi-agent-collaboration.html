Multi-Agent 협업 실전:
에이전트들이 서로 충돌할 때

두 에이전트가 같은 재고를 두고 동시에 발주했다. 합산 금액이 예산의 2배였다.

박
박성훈
2026년 6월 약 3,500자 · 13분 읽기 ◈ 실전 케이스 포함

사고가 났다. 그것도 두 에이전트가 동시에.

공급망 에이전트 A가 부품 재고 부족을 감지하고 1,200만 원어치 발주를 냈다. 같은 시각, 물류 에이전트 B도 동일한 재고 부족을 확인하고 800만 원어치 발주를 냈다. 둘 다 옳았다. 둘 다 자신의 World Model 안에서 완벽하게 합리적인 결정을 내렸다.

문제는 서로의 결정을 몰랐다는 것이다.

합산 발주 금액: 2,000만 원. 실제 필요량: 700만 원어치. 창고에는 갑자기 재고가 넘쳐흘렀고, 나머지 1,300만 원은 3개월 뒤에야 소진됐다.

왜 에이전트끼리 충돌하는가

단일 에이전트 시스템에서는 World Model이 하나다. 에이전트 하나가 세상을 이해하고, 하나가 결정을 내린다.

Multi-Agent 시스템에서는 World Model이 에이전트 수만큼 존재한다. 각 에이전트는 자신만의 세계 이해를 가지고 있고, 그 이해가 서로 다를 수 있다.

● 충돌 발생 구조 
에이전트 A의 World Model:
  재고: 부품-K7 잔량 12개 → 부족 (임계치 50개)
  예측: 3일 후 생산 라인 중단 위험
  결정: 즉시 발주 ₩1,200만

에이전트 B의 World Model:
  재고: 부품-K7 잔량 12개 → 부족 (임계치 30개)
  예측: 5일 후 물류 지연 위험
  결정: 즉시 발주 ₩800만

공유된 정보: 없음
충돌 감지: 없음 → 중복 발주 실행

두 에이전트의 World Model이 같은 현실(재고 12개)을 보면서도 다른 임계치, 다른 예측, 다른 결정으로 이어졌다. 각자의 관점에서는 완벽히 올바른 행동이었다.

이것이 World Model 충돌(World Model Conflict)이다. 에이전트 개별 성능이 아무리 높아도, 시스템 수준에서 일관성이 없으면 반드시 발생한다.

Multi-Agent 충돌의 세 가지 유형

HAchillesWorld가 분류한 Multi-Agent 충돌 패턴은 크게 세 가지다.

1. 상태 충돌 (State Conflict)

여러 에이전트가 동일한 자원의 상태를 각자 다르게 인식한다. 위의 발주 사례가 전형적인 예다. 재고라는 단일 사실을 두 에이전트가 서로 다른 World Model로 읽는다.

2. 계획 충돌 (Plan Conflict)

각 에이전트의 계획이 다른 에이전트의 계획을 방해한다. 에이전트 A가 특정 서버 리소스를 점유하는 계획을 세우는 동안, 에이전트 B가 같은 리소스를 사용하는 작업을 스케줄링한다.

3. 목표 충돌 (Goal Conflict)

가장 근본적인 충돌. 에이전트 A의 목표(비용 최소화)와 에이전트 B의 목표(납기 준수)가 서로 상충될 때, 둘 다 자신의 목표를 최적화하면서 시스템 전체의 성과를 악화시킨다.

충돌 유형 발생 빈도 비용 영향 감지 난이도
상태 충돌 높음 즉각적 중간
계획 충돌 중간 지연됨 어려움
목표 충돌 낮음 누적·만성 매우 어려움

HAchillesWorld로 Multi-Agent 시스템 진단

단일 에이전트 진단과 Multi-Agent 진단의 차이는 시스템 수준 일관성 지표의 유무다.

multi_agent_scan.py 
from hachillesworld import HAchillesWorldClient, instrument

client = HAchillesWorldClient()

# 각 에이전트에 동일한 클라이언트 인스턴스로 계측
@instrument(client, agent_name="supply-agent")
class SupplyAgent: ...

@instrument(client, agent_name="logistics-agent")
class LogisticsAgent: ...

# Multi-Agent 시스템 전체 진단
report = client.scan_multi_agent(
    agents=["supply-agent", "logistics-agent"],
    logs="./system_logs.json"
)
print(report.conflict_summary())

💡 권장: Shared World Model 레이어 구축

개별 에이전트는 80점대로 양호하지만 시스템 일관성이 34점이다. 이 숫자가 핵심이다. 에이전트를 개별적으로 아무리 잘 만들어도 시스템 수준 일관성이 없으면 충돌은 불가피하다.

해결책: Shared World Model 레이어

Multi-Agent 충돌의 근본 원인은 공유되지 않은 현실 이해다. 해결 방향은 하나다: 에이전트들이 공통의 현실 인식을 갖도록 만드는 것.

HAchillesWorld v2.1이 제공하는 SharedWorldModel은 에이전트 간에 중요한 상태 정보를 실시간으로 동기화하는 레이어다.

shared_world_model.py 
from hachillesworld import HAchillesWorldClient, SharedWorldModel

client = HAchillesWorldClient()

# 공유 World Model 초기화 — 모든 에이전트가 참조
swm = SharedWorldModel(
    client=client,
    sync_interval_ms=500,   # 0.5초마다 동기화
    conflict_policy="latest-wins"
)

# 에이전트가 행동 전 공유 상태를 읽고 쓴다
class SupplyAgent:
    def plan(self, state, goal):
        # 발주 전 다른 에이전트의 의도를 확인
        pending = swm.get("pending_orders")
        if pending.covers(state.shortage):
            return Action.SKIP   # 중복 발주 방지

        # 발주 의도를 공유 상태에 먼저 등록
        swm.set("pending_orders", {
            "agent": "supply-agent",
            "item": state.shortage.item,
            "amount": goal.order_qty,
            "expires_at": now() + timedelta(minutes=5)
        })
        return Action.ORDER

핵심 패턴은 "행동 전 의도 등록, 행동 전 의도 확인"이다. 에이전트가 중요한 행동을 하기 전에 공유 공간에 자신의 의도를 먼저 기록하고, 다른 에이전트의 의도도 확인한 뒤 행동한다.

Harness 기반 에이전트 간 조율

Shared World Model은 상태를 공유하지만, 행동 규칙은 별도로 관리해야 한다. HAchillesWorld의 Multi-Agent Harness가 이를 담당한다.

● Multi-Agent Harness 규칙 예시 
규칙 MA-001: 발주 전 공유 상태 확인 필수
  조건: action.type == "ORDER"
  실행: check swm["pending_orders"] before proceed

규칙 MA-002: 동일 자원 동시 접근 금지
  조건: resource.concurrent_agents > 1
  실행: queue 처리, 선착순 잠금

규칙 MA-003: 예산 소진 시 타 에이전트 알림
  조건: budget.remaining < threshold
  실행: broadcast to all agents

규칙 MA-004: 충돌 감지 시 인간 승인 요청
  조건: conflict.severity == "HIGH"
  실행: pause + notify operator

이 규칙들은 코드가 아니라 선언형 정책으로 작성된다. 에이전트 코드를 수정하지 않아도 Harness 규칙만 추가하면 충돌 방지 로직이 적용된다.

Multi-Agent 시스템 도입 전 체크리스트

Multi-Agent 시스템을 새로 구축하거나 기존 시스템에 에이전트를 추가할 때 반드시 확인해야 할 항목이다.

  • 공유 자원 목록 작성 — 둘 이상의 에이전트가 접근하는 모든 자원을 식별한다.
  • 임계치 통일 — 에이전트마다 다른 판단 기준(임계치)을 하나의 설정 파일로 통합한다.
  • 의도 공유 메커니즘 — 중요한 행동 전에 의도를 공유 공간에 등록하는 패턴을 설계한다.
  • 충돌 감지 모니터링 — HAchillesWorld 시스템 일관성 점수를 지속 관측한다.
  • 충돌 정책 문서화 — 충돌 발생 시 어떤 에이전트의 결정이 우선하는지 사전에 정의한다.
  • 인간 개입 조건 설정 — 자동 처리 한계를 벗어난 충돌에 대한 알림과 승인 흐름을 만든다.
