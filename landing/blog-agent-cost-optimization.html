AI 에이전트 비용 최적화
실전 가이드

LLM 호출 70%를 줄이고 연간 ₩3,600만을 절감한 구체적인 방법

박
박성훈
2026년 6월 약 3,200자 · 12분 읽기 ◈ 실전 케이스 포함

지난 분기, 우리 팀의 AI 에이전트 운영 비용 청구서를 처음 열었을 때 나는 할 말을 잃었다.

월 LLM API 비용 ₩1,280만 원. 예상의 세 배였다.

처음엔 트래픽이 늘었다고 생각했다. 틀렸다. 로그를 뒤져보니 에이전트가 같은 질문을 평균 7.3번 반복 호출하고 있었다. 답을 얻은 뒤에도 확인을 위해 다시 물어보고, 또 물어봤다. 자신감이 없어서였다.

이것이 비용 문제의 본질이었다. 에이전트가 세상을 제대로 이해하지 못할수록 더 많이 물어본다.

AI 에이전트 비용의 실제 구조

일반적으로 AI 에이전트 운영 비용을 이렇게 상상한다: "질문 하나 → LLM 호출 한 번 → 답변." 하지만 현실은 훨씬 복잡하다.

● 에이전트의 실제 LLM 호출 패턴 (로그 분석) 
태스크 1건당 평균 호출 분해:

  계획 수립        : 1.0회  (필수)
  상태 재확인      : 2.3회  ← World Model 불신
  중간 검증        : 1.8회  ← 과신 보정 부재
  오류 후 재시도   : 2.2회  ← Harness 미작동
  ──────────────────────────
  합계             : 7.3회  (이상적: 1.5회)

HAchillesWorld 진단이 밝혀낸 비용 낭비의 세 가지 근원:

  • World Model Drift: 환경이 바뀌었는데 에이전트는 과거 정보로 계획을 세운다. 예측이 틀리면 재호출한다.
  • 얕은 Planning Depth: 몇 스텝 앞을 못 보기 때문에 매 스텝마다 새로 물어봐야 한다.
  • Harness 부재: 이미 배운 실수를 기억하지 못해서 같은 오류를 반복하고 재시도한다.

HAchillesWorld로 비용 낭비 지점 진단

먼저 현재 에이전트의 비용 구조를 수치로 확인해야 한다. HAchillesWorld Scan이 이것을 자동으로 분해한다.

bash 
pip install hachillesworld
hachillesworld scan \
  --logs ./agent_logs.json \
  --name "my-agent" \
  --cost-analysis

Scan 결과에서 주목해야 할 세 가지 지표:

● HAchillesWorld 비용 진단 리포트 
⚠ 비용 효율성 분석

  LLM 호출 효율      : 21%    🔴  (이상치: 70% 이상)
  Simulation Drift    : 34%    🔴  (목표: 5% 미만)
  Calibration ECE     : 0.27   🔴  (목표: 0.10 미만)
  Harness Coverage    : 4개    ⚠   (권장: 20개 이상)

  💡 예상 절감 가능액: ₩890만/월 (현재 대비 -69%)

LLM 호출 효율 21%는 호출의 79%가 낭비라는 뜻이다. Calibration ECE 0.27은 에이전트가 "90% 확신"을 말할 때 실제 정확도가 63%임을 의미한다 — 자신이 없어서 계속 재확인하는 구조다.

Planning Depth를 높여 선제 계획하기

비용 절감에서 가장 극적인 효과를 낸 것은 Planning Depth 개선이었다.

Depth가 낮은 에이전트는 근시안적이다. 3스텝 앞밖에 못 보면, 4스텝 후에 문제가 생길 때 그 시점에서 다시 LLM을 호출해야 한다. Depth 15라면, 처음 계획을 세울 때 15스텝의 결과를 미리 시뮬레이션해서 중간에 재호출할 필요가 없다.

Planning Depth 태스크당 LLM 호출 수 월 비용 (동일 트래픽) 상태
3 (초기) 7.3회 ₩1,280만 낭비
8 (Phase 1) 3.9회 ₩680만 개선 중
15 (Phase 2) 2.1회 ₩370만 양호
22 (Phase 3) 1.6회 ₩280만 최적화

Planning Depth는 MCTS(Monte Carlo Tree Search)를 기반으로 단계적으로 끌어올린다. HAchillesWorld Optimize 명령이 현재 Depth에서 다음 단계로 올리기 위한 구체적인 로드맵을 자동 생성한다.

bash 
# 비용 최적화 로드맵 자동 생성
hachillesworld optimize ./scan_report.json \
  --target cost \
  --weeks 12

Harness로 반복 실수 차단

두 번째로 큰 절감 효과는 Harness에서 나왔다.

Harness는 에이전트가 이미 경험한 실수를 다시 반복하지 않도록 막는 규칙 집합이다. 규칙이 없으면 에이전트는 오류가 날 때마다 LLM에게 "이건 어떻게 처리해야 하나요?"를 반복 질문한다.

초기 우리 에이전트의 Harness Coverage는 4개였다. 가장 빈번하게 발생하는 오류 패턴 20개를 Harness 규칙으로 등록했다. 결과는 즉각적이었다.

● Harness 적용 전후 오류 재시도 비율 
오류 유형                  | 적용 전  | 적용 후
──────────────────────────────────────────────
재고 부족 처리              | 재호출 3.1회 | 즉시 처리 (0회)
외부 API 타임아웃           | 재호출 2.8회 | 자동 폴백 (0회)
예산 초과 경계 케이스       | 재호출 4.2회 | 규칙 적용 (0회)
중복 발주 감지              | 재호출 1.9회 | Harness 차단 (0회)
──────────────────────────────────────────────
절감된 호출 수 (월 기준)   |          | 약 1,840회

Phase 3에서는 Meta-Harness가 새로운 오류 패턴을 자동으로 감지해서 규칙을 스스로 생성했다. 8개의 자동 생성 규칙이 추가됐고, 모두 승인했다. 이제 에이전트는 더 영리해질수록 스스로 더 저렴해진다.

과신(Overconfidence) 교정으로 재확인 호출 제거

가장 놓치기 쉬운 비용 원인이 Calibration이다.

에이전트가 "85% 확신"이라고 말할 때 실제로 85% 정확하다면, 재확인이 필요 없다. 하지만 ECE(Expected Calibration Error)가 0.27인 에이전트는 85% 확신할 때 실제 정확도가 58%다. 이 격차를 에이전트 스스로 느끼기 때문에 계속 재확인한다.

Temperature Scaling 적용으로 ECE를 0.27 → 0.06으로 낮췄다. 코드 20줄이었다.

calibrate.py 
from hachillesworld import HAchillesWorldClient

client = HAchillesWorldClient()

# Calibration 자동 측정
report = client.calibrate(
    agent_name="my-agent",
    logs="./agent_logs.json"
)
print(f"ECE: {report.ece:.3f}")  # 0.270

# Temperature Scaling 적용
client.apply_temperature_scaling(
    agent_name="my-agent",
    target_ece=0.08
)

ECE가 낮아지자 에이전트는 확신할 때 실제로 확신하기 시작했다. 재확인 호출이 62% 감소했다.

비용이 줄었을 뿐만 아니라 성능도 올라갔다. LLM 호출을 줄이기 위해 에이전트가 더 정확해졌기 때문이다. 비용 최적화와 성능 향상이 같은 방향이다.

"에이전트를 더 저렴하게 만드는 방법은 더 영리하게 만드는 것이다."
