2024년 초, 한 스타트업이 LLM 기반 고객 응대 에이전트를 프로덕션에 배포했다. LangSmith로 토큰 사용량을 추적했고, Datadog으로 응답 시간을 모니터링했다. 대시보드에는 초록 불이 가득했다.

배포 3일 뒤, 에이전트는 실제로 존재하지 않는 환불 정책을 발명해서 2,000여 명의 고객에게 안내했다. 법무팀이 개입해야 했다.

기존 도구들은 아무것도 감지하지 못했다. 토큰 사용량은 정상이었고, 응답 시간도 빨랐으며, 에러 로그는 비어 있었다. 에이전트는 기술적으로 완벽하게 작동하고 있었다. 단지, 세상을 잘못 이해하고 있었을 뿐이다.

이 포스트는 AI 에이전트 진단도구가 어떻게 발전해 왔는지, 그리고 기존 도구들이 놓치는 것이 무엇인지를 정리한다.

진단도구의 세대별 발전

AI 모니터링 도구의 역사는 세 세대로 나눌 수 있다. 각 세대는 당시의 AI 기술 수준에 적합한 도구였다. 문제는 AI가 에이전틱으로 진화했는데 도구는 여전히 이전 세대에 머물러 있다는 것이다.

1세대 (2020~)
로그 수집 · 비용 추적
  • 토큰 사용량 측정
  • API 호출 횟수·비용 집계
  • 에러 로그 수집
  • 응답 시간(latency) 추적
⚠ 한계 "무슨 일이 일어났는가"는 알지만 "왜"는 모른다
2세대 (2022~)
실험 추적 · 품질 평가
  • 프롬프트 버전 관리
  • LLM 출력 품질 자동 평가 (LLM-as-judge)
  • 할루시네이션 감지
  • A/B 테스트, 실험 비교
⚠ 한계 단일 응답을 평가하지만 에이전트의 계획·행동 루프는 평가 못 함
3세대 (2026~)
World Model 진단
  • 에이전트가 세상을 얼마나 잘 이해하는지 측정
  • Planning Depth — 몇 단계 앞까지 시뮬레이션하는가
  • Simulation Drift — 예측이 현실과 얼마나 벌어지는가
  • Calibration ECE — 자신의 확신도가 정확한가
  • 15개 지표 → HAS 종합 점수 → 레벨 판정
◈ HAchillesWorld "왜 그 결정을 내렸는가"와 "다음엔 어떻게 막는가"를 답함

기존 도구의 3가지 구조적 한계

LangSmith, Weights & Biases, Datadog AI, Arize는 각각 훌륭한 도구다. 문제는 이 도구들이 단일 LLM 호출을 모니터링하기 위해 설계되었다는 것이다. 에이전트는 다르다.

01

WHAT에는 답하지만 WHY에는 답하지 못한다

기존 도구는 "에이전트가 토큰 1,240개를 사용했고, 응답 시간은 3.2초였으며, 에러가 없었다"고 알려준다. 그러나 에이전트가 왜 그 행동을 했는지, 그 결정의 내부 논리가 현실에 맞는지는 알려주지 않는다. 실패가 발생하면 로그를 뒤져 원인을 추론해야 한다.

02

단일 응답을 평가하지만 연쇄 계획은 평가 못 한다

LLM-as-judge 방식은 단일 응답의 품질을 평가한다. 그러나 에이전트의 핵심은 여러 단계에 걸친 계획(Planning)이다. 1단계 행동이 3단계 후의 결과에 어떤 영향을 미치는지, 에이전트가 얼마나 앞을 내다보고 결정하는지는 단일 응답 평가로는 파악할 수 없다.

03

증상은 감지하지만 근본 원인은 진단 못 한다

에이전트 비용이 예산을 143% 초과했다. Datadog은 이것을 알람으로 알려준다. 그러나 왜 초과했는지는 알려주지 않는다. HAchillesWorld의 진단 결과는 "Planning Depth 8 — 단기 반응만 가능, 발주 리드타임(10일)을 시뮬레이션 범위 밖에서 결정하고 있음"이었다. 근본 원인이 다르면 처방도 달라야 한다.

에이전트 실패의 77%는 코드 버그나 API 에러가 아니라 World Model 문제에서 발생한다. 기존 도구는 이 77%를 감지하지 못한다.

패러다임 전환: What에서 Why로

기존 도구와 HAchillesWorld의 근본적인 차이는 측정 대상이 다르다는 것이다.

  • 기존 도구는 에이전트의 행동 결과를 측정한다. 토큰을 얼마나 썼는가, 얼마나 빨리 응답했는가, 에러가 몇 번 났는가.
  • HAchillesWorld는 에이전트의 인지 구조를 측정한다. 에이전트가 세상을 얼마나 정확하게 모델링하고 있는가, 얼마나 앞을 내다보고 결정하는가, 자신의 판단에 대한 확신이 근거 있는가.

자동차에 비유하면, 기존 도구는 속도계와 연료 게이지다. 현재 속도와 남은 연료를 알려준다. HAchillesWorld는 차량 진단 시스템(OBD)이다. 엔진 압축비, 연소 효율, 센서 이상을 분석해서 "왜 연비가 나쁜지"를 알려준다.

둘 다 필요하다. 속도계 없이는 운전할 수 없다. 그러나 차가 왜 느린지 알려면 속도계만으로는 부족하다.

어디서부터 시작할 것인가

기존 LangSmith, Datadog, W&B를 당장 버릴 필요는 없다. 이 도구들은 운영 모니터링에 여전히 유용하다. HAchillesWorld는 이 도구들이 보지 못하는 영역을 보완한다.

실용적인 접근 방법은 다음과 같다.

  1. 먼저 진단부터 — HAchillesWorld로 현재 에이전트의 HAS 점수와 레벨을 파악한다. 15개 지표 중 무엇이 가장 취약한지 확인한다.
  2. 상위 3개 액션에 집중 — 진단 결과에서 제시하는 액션 아이템 중 HAS 상승폭이 가장 큰 것 3개를 먼저 적용한다.
  3. 4주 후 재진단 — 개선이 실제로 이루어졌는지 수치로 확인한다. Planning Depth가 올랐는지, Calibration ECE가 내려갔는지.
$ 첫 진단 실행 
# 설치
pip install hachillesworld

# 10분 안에 첫 진단
hachillesworld scan \
  --logs ./agent_logs.json \
  --name "my-agent" \
  --domain supply_chain

# 또는 웹 UI로
https://hachillesworld.ai/scan

마치며

AI 에이전트는 단순한 LLM 래퍼가 아니다. 에이전트는 환경을 인식하고, 계획을 세우고, 행동하고, 결과를 관찰하고, 다음 행동을 수정한다. 이 루프 전체를 올바르게 평가하려면 응답 품질 측정을 넘어 인지 구조 진단이 필요하다.

기존 도구들이 나쁜 것이 아니다. 그 도구들이 설계된 시대의 문제에 맞게 설계되었을 뿐이다. AI가 에이전틱으로 진화한 지금, 진단 도구도 함께 진화해야 한다.

HAchillesWorld 이전 세계에서 에이전트 실패는 "로그를 보니까 이런 에러가 있었네요"로 끝났다. HAchillesWorld 이후 세계에서 에이전트 실패는 "Planning Depth가 8이어서 10일치 리드타임을 시뮬레이션 못 하고 있었고, 이걸 15로 올리면 됩니다"로 끝난다.

측정할 수 없으면 개선할 수 없다. 올바른 것을 측정해야 올바른 방향으로 개선된다.
