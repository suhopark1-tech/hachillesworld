플랫폼 용어 완전 해설

HAS·WMQ·ALM·OHM·Levels × Laws… 처음 보면 낯선 용어들의 의미와 연결 관계를 한 글에서 완전히 정리합니다.

박
박성훈
2026년 6월 약 4,800자 · 18분 읽기 ◈ 레퍼런스 가이드

HAchillesWorld를 처음 접하면 낯선 약어들이 눈에 들어온다. HAS가 90이라는 게 무슨 뜻인지, WMQ와 ALM이 왜 따로 존재하는지, L2와 L3의 차이가 실제로 무엇인지 — 한 번에 명확히 정리하기 어렵다.

이 글은 플랫폼 전체를 관통하는 핵심 용어 20개를 개념 → 측정 방법 → 해석 방법 순서로 설명한다. 북마크해 두고, 리포트를 볼 때마다 참고하면 된다.

종합 지표 — HAS와 세 카테고리

HAchillesWorld의 모든 측정은 15개 세부 지표 → 3개 카테고리 점수 → 1개 종합 점수로 집계된다. 트리 구조를 이해하면 리포트 해석이 훨씬 명확해진다.
종합 지표
HAS — HAchilles Score
하킬레스 종합 점수 · 0~100

에이전트 배포 준비도를 하나의 숫자로 표현한 HAchillesWorld의 핵심 지표다. 세 카테고리(WMQ · ALM · OHM)의 가중 평균으로 계산된다.

HAS = WMQ × 0.45 + ALM × 0.30 + OHM × 0.25

해석 기준: 90 이상(A+) 즉시 배포 가능 · 80~89(A) 조건부 배포 · 70~79(B) 제한적 운영 · 60~69(C) 개선 필요 · 60 미만(D) 배포 보류.

HAW-STUDY-001 실증 연구에서 HAS와 실제 비즈니스 성과 간 상관계수 ρ = 0.9948이 측정됐다. 24주 개선 세션에서 HAS 670→910 상승 시 연간 ₩4,400만 절감 효과가 확인됐다.

가중치(0.45 / 0.30 / 0.25)는 실증 데이터 축적에 따라 버전별로 업데이트되며, 시계열 비교가능성을 위해 버전 정보가 리포트에 함께 기록된다.

카테고리 A
WMQ — World Model Quality
월드 모델 품질 · 0~100

에이전트가 세상을 얼마나 정확하게 이해하고 예측하는지를 측정한다. HAS 가중치의 45%를 차지하는 가장 중요한 카테고리다.

세부 지표 5개: PER(예측 오차율) · ECE(보정 오차) · SDR(시뮬레이션 표류율) · ODR(이상 탐지율) · PD(계획 깊이).

WMQ가 낮으면 에이전트는 현실과 다른 세상을 기반으로 의사결정을 내린다. 코드와 인프라가 완벽해도 WMQ가 낮은 에이전트는 반드시 실패한다.

카테고리 B
ALM — Agency Level Metrics
에이전시 수준 지표 · 0~100

에이전트가 얼마나 자율적이고 유연하게 행동할 수 있는지를 측정한다. HAS 가중치의 30%.

세부 지표 5개: SCR(자기 수정율) · CA(반사실 정확도) · GAR(목표 달성율) · AS(적응 속도) · HC(계층 일관성).

ALM이 낮은 에이전트는 예상치 못한 상황에서 방향을 잃는다. "잘 작동하다가 갑자기 이상한 결정을 내린다"는 증상의 대부분은 ALM 문제에서 비롯된다.

카테고리 C
OHM — Operational Health Metrics
운영 건전성 지표 · 0~100

에이전트가 프로덕션 환경에서 얼마나 안정적으로 운영되는지를 측정한다. HAS 가중치의 25%.

세부 지표 5개: WMUL(월드 모델 갱신 지연) · IRT(장애 복구 시간) · HITL(사람 개입 비율) · HR(할루시네이션 비율) · SU(시스템 활용률).

OHM은 장기 운영의 신뢰성을 결정한다. 초기에 잘 작동했던 에이전트가 3개월 후 성능이 떨어지는 이유가 여기에 있다.

분류 체계 — Levels × Laws

HAchillesWorld는 모든 에이전트를 역량 레벨(Levels)과 운영 도메인(Laws)의 2차원 좌표로 배치한다. 이 좌표가 리포트 해석과 개선 목표 설정의 기준이 된다.
분류 체계
Levels — 에이전트 역량 레벨
L1 Predictor · L2 Simulator · L3 Evolver

에이전트가 미래를 얼마나 멀리, 얼마나 복잡하게 시뮬레이션할 수 있는지를 나타낸다.

L1 Predictor — 패턴 매칭과 단일 스텝 예측. 현재 상태에서 한 걸음 앞을 본다. 재고 경보, 이상 탐지 등 단순 반응형 작업에 적합하다.

L2 Simulator — 멀티스텝 롤아웃과 계획 수립. 3~10스텝 앞을 시뮬레이션하며 최적 경로를 찾는다. 공급망 최적화, 일정 계획 등에 적합하다.

L3 Evolver — 환경 변화에 자율 적응. 새로운 규칙을 스스로 학습하고 World Model을 갱신한다. 가장 복잡한 동적 환경에 배치된다.

리포트의 L2.3과 같은 표기는 "L2 레벨 내에서 30% 진행됐다"는 뜻이다. 소수점은 0~9 스케일의 레벨 내 진행도다.

분류 체계
Laws — 운영 도메인 법칙
Digital · Physical · Social · Scientific

에이전트가 작동하는 세계의 물리적·논리적 규칙 체계다. 같은 레벨의 에이전트라도 Laws에 따라 완전히 다른 진단 기준이 적용된다.

Digital Laws — API·소프트웨어·데이터 처리. 결정론적 규칙과 명확한 피드백이 특징이다. 대부분의 엔터프라이즈 에이전트가 여기에 속한다.

Physical Laws — 로봇·제조·물리 환경. 뉴턴 역학과 실시간 센서 피드백이 지배한다. 지연과 불확실성이 가장 높다.

Social Laws — 멀티에이전트·협상·고객 대응. 언어 화용론과 사회 규범이 작동한다. 상황 의존성이 극도로 높다.

Scientific Laws — 연구·분석·가설 검증. 실험 설계와 통계적 추론이 핵심이다. 장기 학습 루프가 특징이다.

WMQ 세부 지표 5개

WMQ 지표
PER — Prediction Error Rate
예측 오차율 · 목표: < 0.15

에이전트가 예측한 결과와 실제 결과 사이의 평균 괴리율이다. WMQ의 가장 기본적인 지표.

PER이 0.15라면 에이전트의 예측이 평균적으로 15% 빗나간다는 뜻이다. 공급망 맥락에서 비용 예측 PER이 0.296이면, 482만 원을 예측했을 때 실제로는 625만 원이 된다는 의미다.

PER = mean(|예측값 − 실제값| / 실제값) < 0.15 → OK · 0.15~0.30 → Warning · >0.30 → Critical
WMQ 지표
ECE — Expected Calibration Error
기대 보정 오차 · 목표: < 0.10

에이전트의 자기 확신과 실제 정확도가 얼마나 일치하는지를 측정한다. ECE = 0은 "80% 확신한다"고 할 때 정말로 80% 맞힌다는 뜻이다.

ECE가 높은 에이전트는 위험하다. 틀릴 때도 자신 있게 말하기 때문이다. "80% 확신한다"고 했는데 실제로 50%만 맞힌다면 ECE는 0.30이다.

PER이 낮아도 ECE가 높으면 에이전트를 신뢰하기 어렵다. 리포트에서 두 지표를 함께 확인해야 한다.

WMQ 지표
SDR — Simulation Drift Rate
시뮬레이션 표류율 · 목표: < 0.20

에이전트가 내부적으로 시뮬레이션하는 미래 상태와 실제 미래가 얼마나 빠르게 벌어지는지를 측정한다. "World Model이 현실로부터 얼마나 빠르게 이탈하는가"의 지표다.

SDR은 특히 L2 이상의 에이전트에게 중요하다. 멀티스텝 계획을 세울 때 초반의 작은 오차가 후반에 눈덩이처럼 커지는 현상을 포착한다.

관련 개념: World Model Drift

WMQ 지표
ODR — OOD Detection Rate
분포 이탈 탐지율 · 목표: > 0.80

에이전트가 학습·운영 데이터 분포를 벗어난 입력(Out-of-Distribution)을 얼마나 잘 감지하는지 측정한다. ODR이 높을수록 에이전트는 "나는 이 상황을 잘 모른다"고 적시에 알린다.

ODR이 낮은 에이전트는 전혀 경험하지 못한 상황에서도 높은 확신으로 잘못된 결정을 내린다. 블랙스완 이벤트에 특히 취약하다.

WMQ 지표
PD — Planning Depth
계획 깊이 · 단위: 스텝 수

에이전트가 의사결정 시 몇 스텝 앞까지 시뮬레이션하는지를 측정한다. 바둑으로 비유하면 "몇 수 앞을 읽는가"다.

PD 1~2: 즉각 반응형. 단순 패턴 매칭. PD 3~5: 단기 계획. 일반 업무 자동화에 충분. PD 6~10: 중기 계획. 공급망·프로젝트 관리에 적합. PD 10+: 전략적 사고. 연구·복잡 시스템에 필요.

MCTS(Monte Carlo Tree Search)를 사용하는 에이전트는 Planning Depth가 훨씬 높다. → MCTS Planning Depth 상세 해설 참조.

ALM · OHM 핵심 지표

ALM 지표
SCR — Self-Correction Rate
자기 수정율 · 목표: > 0.70

에이전트가 실행 중 오류를 스스로 감지하고 경로를 수정한 비율이다. SCR이 높을수록 사람의 개입 없이 스스로 궤도를 유지한다.

SCR은 에이전트의 "메타인지" 능력을 반영한다. 틀렸음을 모르는 에이전트는 SCR이 낮다. L3 에이전트의 핵심 역량이다.

ALM 지표
CA — Counterfactual Accuracy
반사실 정확도 · 목표: > 0.65

"만약 X가 달랐다면 결과가 어떻게 됐을까?"라는 반사실 질문에 에이전트가 얼마나 정확하게 답하는지를 측정한다.

CA가 높은 에이전트는 인과관계를 이해하고, 단순 상관관계에 속지 않는다. 전략적 의사결정을 맡길 때 특히 중요하다.

OHM 지표
WMUL — World Model Update Latency
월드 모델 갱신 지연 · 목표: < 300초

환경이 변화했을 때 에이전트의 내부 World Model이 그 변화를 반영하기까지 걸리는 시간이다. 시장이 급변하거나 외부 API 스펙이 바뀔 때 에이전트가 얼마나 빨리 적응하는지를 나타낸다.

WMUL이 길수록 에이전트는 구식 정보로 결정을 내린다. 실시간성이 중요한 트레이딩·물류 에이전트에서 특히 치명적이다.

OHM 지표
IRT — Incident Recovery Time
장애 복구 시간 · 단위: 분

에이전트가 비정상 상태(임계값 위반, 목표 달성 실패, 연속 오류)에서 정상 상태로 복구하는 데 걸리는 평균 시간이다.

IRT는 에이전트의 회복 탄력성(Resilience)을 나타낸다. IRT가 낮을수록 서비스 연속성이 높다.

OHM 지표
HITL — Human-in-the-Loop Rate
사람 개입 비율 · 목표: 역할에 따라 다름

전체 의사결정 중 사람이 개입한 비율이다. HITL이 낮다는 것이 반드시 좋은 것은 아니다 — 역할에 따라 적정 수준이 다르다.

고위험 결정(의료·금융·법률)에서는 HITL이 높은 것이 설계 목표일 수 있다. 반면 반복적 데이터 처리에서 HITL이 높다면 에이전트가 제 역할을 못 하고 있다는 신호다.

리포트에서는 역할 기준 대비 HITL 수준을 비교해 해석해야 한다.

핵심 개념

개념
World Model Drift
월드 모델 표류

에이전트의 내부 World Model이 시간이 지남에 따라 현실로부터 점차 멀어지는 현상이다. 코드 버그나 인프라 장애와 달리 조용히, 서서히 진행되기 때문에 감지하기 어렵다.

Drift의 주요 원인: ① 데이터 분포 변화(Data Drift) ② 외부 환경 규칙 변화 ③ 누적된 예측 오류의 피드백 ④ 모델 가중치의 점진적 변화.

HAchillesWorld의 SDR과 WMUL이 Drift를 실시간으로 추적한다. Operate 모듈은 Drift가 임계값을 초과하면 자동으로 알림을 발송한다.

데이터 단위
Episode
에피소드 — 에이전트 행동의 기본 단위

에이전트가 하나의 목표를 달성하기 위해 수행한 일련의 행동 시퀀스다. Observe(관찰) → Plan(계획) → Execute(실행) → Reflect(반성)의 사이클이 한 Episode를 구성한다.

HAchillesWorld의 모든 지표는 Episode 단위로 측정된다. Episode가 충분히 쌓여야 통계적으로 신뢰할 수 있는 진단이 가능하다. 최소 권장 샘플: 50 Episodes.

SDK의 EpisodeRecord와 LogCollector가 Episode 데이터를 자동으로 수집한다.

아키텍처
Harness
하네스 — 에이전트 테스트 프레임워크

에이전트를 통제된 환경에서 실행하고 성능을 측정하는 테스트 프레임워크다. 자동차 충돌 테스트대처럼, 에이전트를 다양한 시나리오에 노출시켜 World Model의 약점을 찾아낸다.

Harness는 Rule-based Judge와 AI Judge(Claude Sonnet)를 혼합해 판정한다. 규칙으로 명확히 판단할 수 있는 것은 Rule Judge가, 맥락과 뉘앙스가 필요한 것은 AI Judge가 담당한다.

Scan 모듈의 진단은 Harness에서 수집된 데이터를 기반으로 한다.

Scan 진단 결과를 바탕으로 Optimize 모듈이 생성하는 단계별 개선 계획이다. 현재 레벨에서 목표 레벨까지 도달하기 위한 구체적 과제와 예상 기간이 포함된다.

각 Phase는 예상 점수 상승폭과 우선 과제 목록을 제시한다. 단순 권고가 아니라 HAW-STUDY-001 실증 데이터 기반의 수치 예측이다.

Multi-Agent Optimizer는 여러 개의 에이전트를 동시에 진단하고, 리소스를 최적 배분하는 조직 수준의 로드맵도 생성한다.

리포트 읽는 법 — 용어 연결 관계

HAchillesWorld 리포트를 처음 받았을 때 권장하는 순서는 다음과 같다.

  1. HAS 종합 점수와 등급 확인 — 배포 가능 여부의 1차 판단
  2. 세 카테고리(WMQ / ALM / OHM) 점수 비교 — 가장 낮은 카테고리가 우선 개선 대상
  3. Critical 지표 목록 확인 — 임계값을 초과한 지표별 즉시 조치 필요
  4. Levels × Laws 좌표 확인 — 현재 레벨이 배치 환경에 적합한지 판단
  5. Optimization Roadmap 검토 — 다음 레벨로 이동하기 위한 구체적 계획 수립
WMQ가 낮으면서 HAS가 높은 경우는 드물다. WMQ는 HAS의 45%를 차지하기 때문이다. 반대로 OHM이 낮아도 WMQ가 매우 높으면 HAS가 상대적으로 선방할 수 있다. 하지만 OHM 문제는 장기 운영에서 반드시 표면화된다.
