## SCAMPER 개요

SCAMPER는 창의적 문제 해결을 위한 7개 질문 축이다.  
각 축은 기존 제품·서비스·프로세스의 요소를 강제로 비틀어 혁신 아이디어를 끌어낸다.

```
S — Substitute  : 대체하라. 이 요소 대신 다른 것을 쓰면?
C — Combine     : 결합하라. 이것과 저것을 합치면?
A — Adapt       : 적응시켜라. 다른 분야의 무엇을 가져올 수 있나?
M — Modify      : 변형하라. 크게/작게/다르게 만들면?
P — Put to use  : 다른 용도로 써라. 전혀 다른 맥락에서 쓴다면?
E — Eliminate   : 제거하라. 빼면 오히려 좋아지는 게 있나?
R — Reverse     : 역전하라. 순서를 뒤집거나 반대로 하면?
```

각 축마다 제품의 핵심 요소들을 분해하여 적용한다:

> **핵심 요소**: 진단 방법(Scan) / 지표체계(15 Metrics) / 프레임워크(Levels×Laws) / 결과물(Report·Code) / 인증(HCP/HCE/HCAN) / 수익모델 / 콘텐츠(Blog·Book) / 기술(MCTS·ECE·Harness) / 타겟(Enterprise AI팀)

# S — SUBSTITUTE (대체하라)

> **핵심 질문**: 지금 쓰고 있는 이 요소를 다른 것으로 바꾸면 어떻게 되는가?

## S1. "World Model"이라는 용어를 대체한다면?

**[지훈]**: 이 질문이 사실 제일 중요합니다. 저번에도 지적했지만 "World Model"은 LeCun·DeepMind가 선점한 기술 용어입니다. 우리가 쓰는 방식과 다릅니다. 대체어를 검토해야 합니다.

**[도현]**: 마케팅 관점에서 후보들을 제시해볼게요.

| 대체 용어 후보 | 뉘앙스 | 장점 | 단점 |
|--------------|--------|------|------|
| **Agent Intelligence Score (AIS)** | 수치화된 지능 평가 | FICO 스코어처럼 직관적 | "Intelligence" 남발 시대 |
| **Agent Readiness Index (ARI)** | 배포 준비도 | 기업 구매자에게 친숙 | 평범함 |
| **Situational Awareness Score** | 상황 인식 | 군사·보안 느낌, 강렬 | 맥락 어색 |
| **Agent Comprehension Quality (ACQ)** | 이해의 질 | 기술적으로 중립 | 기억하기 어려움 |
| **Operational World Fidelity (OWF)** | 운영 현실 충실도 | 기술적으로 정확 | 너무 학술적 |
| **Reality Gap Score** | 현실과의 격차 | 직관적, 문제가 보임 | 부정적 뉘앙스 |

**[지훈]**: 저는 **"Agent Situational Awareness (ASA)"** 를 추천합니다. 군사 AI·자율주행 커뮤니티에서 이미 쓰는 개념이고, "세상을 얼마나 이해하는가"를 정확히 포착합니다. "World Model"보다 기술적으로 방어하기 쉽습니다.

**[민서]**: 비즈니스 관점에서는 **"Agent Intelligence Audit"** 가 낫습니다. "Audit"은 기업에서 예산을 쓰는 행위에 익숙한 단어입니다. 재무감사, 보안감사처럼 "AI 에이전트 감사"는 이사회 보고가 가능한 언어입니다.

**[도현]**: 절충안: 외부 마케팅은 "Agent Intelligence Audit", 기술 문서에서는 "World Model Quality"를 유지. 일반인 언어와 전문가 언어를 분리합니다.

> **💡 아이디어 S1**: 마케팅 레이어에서 "Agent Intelligence Audit" 브랜딩, 기술 레이어에서 "World Model Quality" 유지. 두 언어를 계층적으로 운용한다.

## S2. Scan 입력 방식을 대체한다면?

**[현아]**: 현재 Scan은 폼에 텍스트를 입력합니다. 실제 현장에서 에이전트를 운영하는 팀은 "내 에이전트를 설명하는 글"을 쓰는 것 자체가 장벽입니다. 대체 입력 방식들:

| 현재 방식 | 대체 방식 | 기대 효과 |
|----------|----------|----------|
| 텍스트 폼 입력 | **GitHub 레포지토리 URL 입력** | 코드를 직접 분석, 훨씬 객관적 |
| 텍스트 폼 입력 | **로그 파일 드래그&드롭** | 실제 운영 데이터 기반 |
| 텍스트 폼 입력 | **LangSmith / Arize 연동** | 기존 툴 사용자 즉시 진입 |
| 텍스트 폼 입력 | **슬랙 봇으로 대화형 인터뷰** | 질문 방식으로 자연스럽게 수집 |
| 텍스트 폼 입력 | **5분짜리 체크리스트 (Yes/No 20개)** | 로그 없어도 가능한 진단 |

**[수진]**: UX 관점에서 가장 매력적인 것은 **GitHub URL 입력**입니다. 개발자는 링크 하나 붙여넣는 것을 가장 쉽게 느낍니다. Snyk, CodeClimate처럼 코드베이스를 분석하는 SaaS가 이미 이 방식으로 개발자를 끌어들이고 있습니다.

**[지훈]**: GitHub 코드 분석은 기술적으로 의미있습니다. 에이전트의 아키텍처 패턴(ReAct, CoT, 멀티에이전트 여부)을 코드에서 추론할 수 있고, 이것이 Level 판정의 첫 번째 신호가 될 수 있습니다.

> **💡 아이디어 S2**: "GitHub URL 한 줄로 에이전트 사전 진단" — 코드 패턴 분석으로 Level 예측값 제공. 정식 Scan의 무료 미리보기로 활용.

## S3. 인증 수수료 모델을 대체한다면?

**[민서]**: 현재 HCP ₩100K, HCE ₩300K, HCAN ₩500K. 이것을 대체하는 모델들:

| 대체 모델 | 설명 | 장점 | 단점 |
|----------|------|------|------|
| **기업 단위 라이선스** | 기업이 팀 전체 인증 일괄 구매 | B2B 계약 단위 커짐 | 개인 직접 구매 사라짐 |
| **구독 포함** | Pro 구독자는 인증 시험 포함 | 구독 전환 유도 | 인증 가치 희석 |
| **성과 기반** | 에이전트 점수 향상 시에만 과금 | 신뢰 극대화 | 측정·분쟁 복잡 |
| **기업 스폰서십** | 기업이 직원 인증비 후원 | 교육 예산 활용 | 판매 사이클 김 |
| **연간 갱신 구독** | ₩100K/년 → 지속 업데이트 | 반복 수익 | 가격 저항 |

**[승규]**: 법적으로 가장 안전한 모델은 **"수료증(Certificate of Completion)"** 모델입니다. 자격(Qualification)이 아닌 수료를 판매하는 것은 민간자격관리법 적용 대상이 아닙니다. Coursera·Udemy가 이 방식으로 운영합니다. "인증"이라는 단어보다 "수료·이수"가 법적으로 안전합니다.

> **💡 아이디어 S3**: 인증 → 수료 리브랜딩 + 기업 팀 패키지(5인 ₩1M 등) 추가. 법적 리스크 감소 + 객단가 상승.

## S4. MCTS 기술을 대체한다면?

**[지훈]**: MCTS는 강력하지만 LLM 에이전트 맥락에서 구현이 복잡합니다. 대체 기술 검토:

| 기술 | 설명 | Planning Depth 효과 | 구현 난이도 |
|------|------|---------------------|------------|
| **MCTS** (현재) | 트리 탐색 기반 계획 | 매우 깊음 | 매우 높음 |
| **ReAct + Reflection** | 사고-행동-관찰 루프 | 중간 | 낮음 |
| **Tree of Thoughts (ToT)** | GPT-4 기반 분기 사고 | 깊음 | 중간 |
| **Chain of Draft** | 초안 반복 개선 | 얕음 | 매우 낮음 |
| **Best-of-N Sampling** | N번 시뮬레이션 후 최선 선택 | 중간 | 낮음 |
| **Beam Search** | 상위 k개 경로 유지 | 깊음 | 중간 |

**[현아]**: 현장에서는 구현 난이도가 결정적입니다. "MCTS 계획기를 도입하세요"라고 하면 팀장이 "몇 명의 엔지니어가 몇 달 필요한가요?"라고 즉시 묻습니다. **ToT(Tree of Thoughts)** 는 GPT-4 API 호출만으로 구현 가능해서 현실적입니다.

> **💡 아이디어 S4**: 사용자 수준별 기술 선택지 제공 — "빠른 구현: ToT (2주)" / "고급: MCTS (8주)". 같은 결과를 다른 경로로 달성하는 메뉴 제공.

## C1. Scan + Operate = 실시간 지속 진단

**[수진]**: 지금은 Scan이 일회성 진단입니다. Operate는 별도 대시보드입니다. 이 둘을 합치면?

```
현재:
  Scan(일회) → 리포트 → [몇 주 후] → Operate(별도)

결합 제안:
  Continuous Scan = Scan + Operate 통합
  → 에이전트가 실행될 때마다 자동으로 World Model 점수 업데이트
  → 실시간 드리프트 경보 → 즉시 재보정 트리거
```

**[지훈]**: 기술적으로 이것은 **온라인 평가(Online Evaluation)** 패러다임입니다. 에이전트 실행 훅(hook)에 HAchillesWorld 인스트루멘테이션을 붙여서 매 실행마다 15개 지표를 업데이트합니다. 이미 `@instrument` 데코레이터 개념이 있으니 자연스러운 확장입니다.

**[민서]**: 비즈니스 모델도 달라집니다. 일회성 Scan 판매 → 월 구독(에이전트 실행 건수 기반 과금). Datadog, New Relic처럼 관찰 가능성(Observability) SaaS 모델입니다.

> **💡 아이디어 C1**: "HAchillesWorld Live" — 에이전트 실행 시마다 자동 지표 업데이트. 기존 @instrument 데코레이터를 확장하면 구현 가능. 월 구독 수익 모델 전환의 핵심.

## C2. Blog + Certification = 학습 경로(Learning Path)

**[도현]**: 지금 블로그 글 3편과 인증 프로그램이 완전히 분리되어 있습니다. 이걸 연결하면:

```
현재:
  Blog (읽기용) ─── (연결 없음) ─── Cert (구매)

결합 제안: Learning Path
  Blog Article 01 → 실습 과제 → Blog Article 02 → 실습 과제
  → 중간 퀴즈 → Blog Article 03 → 최종 프로젝트 → HCP 수료
```

**[수진]**: UX로 구현하면 각 블로그 글 하단에 "이 글이 HCP 커리큘럼의 Module 1입니다. 다음 모듈 보기 →" 버튼을 추가합니다. 블로그 독자가 자연스럽게 수료 과정으로 유입됩니다.

**[도현]**: 콘텐츠 마케팅의 전형적인 실수가 "읽고 끝"입니다. 블로그가 인증으로 이어지는 구조를 만들면 전환율이 3~5배 오릅니다. HubSpot Academy가 이 모델로 성공했습니다.

> **💡 아이디어 C2**: 블로그 글을 커리큘럼 모듈로 재구성. 각 글 하단에 "HCP Module N" 배지 + 다음 모듈 링크 추가. 기존 콘텐츠 자산을 학습 퍼널로 전환.

## C3. HAchillesWorld + EU AI Act 컴플라이언스 결합

**[승규]**: 2026년 EU AI Act가 본격 시행됩니다. High-risk AI 시스템은 투명성·견고성·정확성을 문서화해야 합니다. HAchillesWorld의 15개 지표 중 다수가 이 요건에 직접 대응합니다.

| EU AI Act 요건 | HAchillesWorld 대응 지표 |
|---------------|------------------------|
| 견고성(Robustness) | Simulation Drift, Self-Correction Rate |
| 정확성(Accuracy) | Calibration ECE, World Model Quality |
| 투명성(Transparency) | Planning Depth, Harness Coverage |
| 인간 감독(Human Oversight) | Harness Coverage, 에이전시 수준 |

**[민서]**: 이것이 엄청난 GTM 기회입니다. "EU AI Act 컴플라이언스 보고서 자동 생성" 기능을 추가하면, HAchillesWorld가 단순 진단 도구를 넘어 **규제 준수 인프라**가 됩니다. 법무팀과 AI팀이 모두 구매 이해관계자가 됩니다.

**[지훈]**: 기술적으로 현재 리포트 포맷에 "EU AI Act Article X 대응 현황" 섹션을 추가하면 됩니다. 완전히 새로운 개발 없이 포지셔닝만 확장 가능합니다.

> **💡 아이디어 C3**: "EU AI Act 준비도 스코어카드" 레이어 추가. 기존 15개 지표를 EU AI Act 조항에 매핑한 별도 리포트 제공. 법무·컴플라이언스 팀을 새 구매자로 추가.

## C4. 책 + 도구 = "책 읽으면 도구가 딸려온다"

**[도현]**: 《Agentic World Modeling 2027》이 출판되면 강력한 자산이 됩니다. 책과 도구를 결합한 상품을 만들 수 있습니다:

```
결합 상품 아이디어:
  Option A — "책 + 1회 무료 Scan 크레딧"
  Option B — "책 + HCP 학습 경로 6개월 접근권"
  Option C — "책 번들 판매" — Kyobo·Yes24 QR코드로 스캔하면 
              독자 전용 Scan 대시보드 접근
```

**[민서]**: 오라일리(O'Reilly) 기술서적의 성공 패턴이 이것입니다. 책이 도구의 가장 긴 세일즈 브로슈어가 됩니다. 책 한 권이 팔릴 때마다 잠재 고객이 생깁니다.

**[현아]**: 현장 교육 구매 담당자 입장에서 "책+도구 패키지"는 예산 집행이 쉽습니다. 교육비 예산으로 처리할 수 있습니다.

> **💡 아이디어 C4**: 책 출판 시 동봉 QR코드 → 독자 전용 Scan 크레딧 제공. 책이 세일즈 퍼널의 입구가 된다.

## C5. Scan + Slack/Teams 봇 결합

**[현아]**: 기업에서 AI 에이전트를 모니터링하는 팀은 이미 Slack을 씁니다. HAchillesWorld를 Slack 봇으로 연동하면:

```
/hachilles status → 현재 에이전트 World Model 점수 조회
/hachilles drift  → 지난 24시간 Drift 변화 보고
/hachilles alert  → ECE 임계치 초과 시 자동 알림
/hachilles report → 주간 요약 리포트 자동 생성
```

**[수진]**: Slack 봇은 설치하면 매일 가시성이 생깁니다. 도구를 굳이 찾아오지 않아도 됩니다. 일일 활성 사용(DAU)이 자연스럽게 만들어집니다. Datadog의 Slack 알림이 이 방식으로 끈끈한 고객 락인(lock-in)을 만들었습니다.

> **💡 아이디어 C5**: Slack/Teams 봇으로 World Model 상태를 채널에 브로드캐스팅. 알림이 곧 마케팅이 된다.

## A1. FICO 신용점수 모델 적용

**[민서]**: FICO 신용점수는 누구나 아는 숫자입니다. 850점 만점, 300~850 범위. 복잡한 재무 데이터를 하나의 숫자로 압축합니다. HAchillesWorld도 이 모델을 적용할 수 있습니다.

```
현재: 15개 지표를 모두 보여줌 → 복잡, 어디에 집중해야 할지 불분명

FICO 적용:
  HAchilles Agent Score (HAS): 0 ~ 1000점
  300 이하: 즉시 개입 필요 (Red Alert)
  300~600: 개선 권고 (Yellow)
  600~800: 운영 가능 (Green)
  800 이상: L3 Certified (Gold)
```

**[도현]**: "당신의 에이전트 HAchilles Score는 742입니다" — 이 한 줄이 마케팅의 전부가 될 수 있습니다. FICO처럼 기업이 채용·계약 시 "에이전트 HAchilles Score 700 이상 요구"라는 기준을 쓰게 되면 생태계가 만들어집니다.

**[지훈]**: 기술적으로도 합리적입니다. 15개 지표의 가중합(weighted sum)으로 단일 스코어를 만들고, 가중치는 도메인별로 다르게 설정합니다.

> **💡 아이디어 A1**: "HAchilles Agent Score (0~1000)" 단일 지표 도입. 복잡한 15개 지표를 유지하되, 대외적으로는 하나의 숫자로 소통. FICO처럼 업계 표준 스코어를 목표로 한다.

## A2. CMMI(역량 성숙도 모델) 구조 차용

**[현아]**: CMMI는 소프트웨어 개발 조직의 역량을 1~5단계로 평가하는 국제 표준입니다. 미국 국방부가 만들었고 전 세계 기업이 씁니다. HAchillesWorld의 L1/L2/L3는 CMMI와 구조적으로 유사합니다.

```
CMMI (소프트웨어 개발)    HAchillesWorld (AI 에이전트)
Level 1: Initial       → L1 Predictor (반응형)
Level 2: Managed       → L1.5 (초기 관리)
Level 3: Defined       → L2 Simulator (시뮬레이션 가능)
Level 4: Quantitative  → L2.5 (정량 관리)
Level 5: Optimizing    → L3 Evolver (자율 진화)
```

**[민서]**: CMMI 인증은 기업이 수억 원을 투자하는 공인 인증입니다. HAchillesWorld가 "AI 에이전트 버전의 CMMI"로 포지셔닝하면 컨설팅 기업·공공기관에서 즉시 알아봅니다.

**[승규]**: 단, CMMI는 ISACA(국제 정보시스템 감사협회)가 관리하는 공인 체계입니다. HAchillesWorld가 "CMMI와 유사한"이라고 하면 괜찮지만, CMMI 명칭을 직접 사용하면 상표 문제가 됩니다. "AI Agent Maturity Model (AAMM)"처럼 독자 명칭이 필요합니다.

> **💡 아이디어 A2**: Levels × Laws 프레임워크를 "AI Agent Maturity Model (AAMM)"으로 브랜딩. CMMI가 만들어 놓은 "성숙도 모델" 시장 인식을 활용하되 독자 명칭으로 차별화.

---

## A3. 의료 임상시험 프레임워크 적용

**[지훈]**: 의료에서 신약은 임상 1상·2상·3상을 통과해야 시장에 나옵니다. AI 에이전트도 실제 환경에 배포하기 전에 단계별 검증이 필요합니다.

```
임상시험 → AI 에이전트 배포 검증

Phase 0: 소수 환경에서 예비 테스트 → Sandbox Validation
Phase 1: 소량 실환경 투입, 안전성 확인 → Shadow Deployment (10%)
Phase 2: 제한적 운영, 효능 측정 → Limited Rollout (30%)
Phase 3: 전체 배포, 장기 모니터링 → Full Production + HAchillesWorld Live
```

**[현아]**: 이것이 실무에서 필요한 언어입니다. "배포 전에 World Model 점수 800 이상을 확인하세요"라고 하면 기업 리스크 관리팀과 대화할 수 있습니다.

**[민서]**: "에이전트 배포 게이팅(Gating) 기준 제공자"로 포지셔닝 가능합니다. CI/CD 파이프라인에 HAchillesWorld 점수 임계치를 통과해야만 프로덕션 배포가 되도록 통합하면, 도구가 인프라의 일부가 됩니다.

> **💡 아이디어 A3**: "Agent Deployment Gate" — HAchillesWorld Score 임계치를 CI/CD 파이프라인에 통합. 점수 미달 에이전트는 자동 배포 차단. DevOps 게이팅 도구로 포지셔닝.

## A4. 보험 인수심사(Underwriting) 모델 적용

**[승규]**: AI 리스크 보험이 빠르게 성장하는 시장입니다. 기업이 AI 에이전트로 인한 손실(잘못된 발주, 고객 이탈 등)에 보험을 들고 싶어합니다. 그런데 보험사는 리스크를 어떻게 평가해야 하는지 모릅니다.

**[민서]**: HAchillesWorld가 **AI 에이전트 보험 인수심사 기준**을 제공하면 어떻게 될까요? 보험사와 파트너십을 맺어:

```
- HAchilles Score 800+ → 보험료 30% 할인
- HAchilles Score 500 미만 → 가입 거절 또는 높은 보험료
- 분기별 재진단으로 보험료 조정
```

**[도현]**: 이 포지셔닝은 매우 강력합니다. 기업 입장에서 "HAchillesWorld 점수를 올리면 보험료가 낮아진다"는 인센티브는 기존 "AI 성능 올리기"와 완전히 다른 차원의 동기입니다.

> **💡 아이디어 A4**: AI 리스크 보험사와 파트너십. HAchilles Score를 보험 인수심사 기준으로 제공. 보험료 할인 인센티브로 에이전트 최적화 동기 부여.

## A5. 투자 신용평가사(Moody's·S&P) 모델 적용

**[민서]**: Moody's, S&P, Fitch는 기업·국가의 신용을 평가합니다. 투자자는 이 점수를 기반으로 투자 결정을 내립니다. 같은 구조가 AI 에이전트에 적용될 수 있습니다.

```
AI Agent Credit Rating by HAchillesWorld:
  HAA (Triple A): 최고 수준, 자율 운영 가능
  HA+/HA/HA-   : 우량, 제한적 감독 필요
  HB+/HB/HB-  : 투기등급, 높은 감독 필요
  HC / HD      : 배포 부적합
```

**[도현]**: VC 투자자들이 AI 스타트업 실사(due diligence)할 때 "에이전트 HAchilles Rating이 뭐냐"고 묻게 되면 게임이 끝납니다. 시장 인프라가 됩니다.

**[지훈]**: 기술적으로도 타당합니다. 현재 15개 지표 → 3개 대분류(WM품질·에이전시·운영건전성) → 종합 등급의 구조가 이미 있습니다.

> **💡 아이디어 A5**: "AI Agent Rating Agency" 포지셔닝. 단기적으로는 불가능하지만, 장기 비전으로 설정. 중간 단계로 "HAchilles Certified" 로고를 제품·서비스에 부착하는 생태계 구축.

## M1. MAGNIFY — 대상을 AI 에이전트에서 모든 AI 시스템으로 확대

**[지훈]**: 현재 HAchillesWorld는 "AI 에이전트"에 집중합니다. 그런데 기업의 AI 시스템은 에이전트만이 아닙니다:

```
확대 가능한 대상:
├── AI 에이전트 (현재)
├── RAG 파이프라인
├── LLM Fine-tuned 모델
├── ML 예측 모델 (비 LLM)
├── 멀티모달 시스템
└── AI 기반 추천 시스템
```

**[민서]**: "에이전트 전문"은 처음에는 차별화지만, 시장이 커지면 "우리 시스템은 에이전트가 아니라서 못 쓴다"가 장벽이 됩니다. TAM(Total Addressable Market)이 확 커집니다.

**[도현]**: 확대 시 포지셔닝: "AI 시스템의 세계 이해 능력을 측정합니다" → 모든 AI가 대상. 그러나 에이전트에서 시작해 검증된 후 확대하는 것이 전략적으로 맞습니다.

> **💡 아이디어 M1**: 에이전트 → AI 시스템 전반으로 점진적 확장 로드맵. v1.0 에이전트, v2.0 RAG+에이전트, v3.0 전체 AI 시스템.

## M2. MAGNIFY — L3 위에 L4·L5를 추가

**[지훈]**: 현재 L1/L2/L3가 최고 단계입니다. L3 Evolver도 여전히 인간 설계 범위 안입니다. 미래를 위해 L4·L5를 정의하면:

```
L4 Innovator: 새로운 환경을 스스로 정의하고 다른 에이전트를 설계
L5 Sovereign: 자율적으로 목표를 갱신하고 에이전트 생태계를 통솔
              (현재는 이론적 수준, 실현 불명확)
```

**[민서]**: L4·L5를 "미래 비전"으로 공개하면 두 가지 효과가 있습니다. 첫째, HAchillesWorld가 AI 발전의 나침반 역할을 합니다. 둘째, L3 달성 고객이 "그 다음"을 계속 원하게 만듭니다.

**[도현]**: "로드맵 공개"는 강력한 마케팅입니다. Tesla가 FSD Level 5 자율주행을 공언하면서 현재 Level 2를 파는 것처럼, HAchillesWorld도 L5 비전을 공언하면서 L3 달성을 팔 수 있습니다.

> **💡 아이디어 M2**: L4·L5를 "COMING SOON" 로드맵으로 공개. 업계 대화에서 선점 효과. L3 고객이 L4를 기다리게 한다.

---

## M3. MINIFY — 15개 지표를 3개로 압축한 "Quick Score"

**[수진]**: 랜딩 페이지를 처음 보는 사람이 "15개 지표"를 들으면 압도됩니다. 입문 경험을 위해 3개로 압축:

```
HAchilles Quick Score (3대 핵심 지표)
  1. Reality Gap   : 에이전트가 현실을 얼마나 정확히 예측하는가 (Drift + ECE)
  2. Lookahead     : 몇 단계 앞을 내다보는가 (Planning Depth)
  3. Guard Rails   : 알려진 실수를 얼마나 막는가 (Harness Coverage)

→ 3개 합산 0~300점 → "Quick Health Score"
→ 상세 진단을 원하면 Full Scan (15개 지표)
```

**[도현]**: "3개로 먼저 맛보고, 15개로 깊이 들어간다" — 인지 부하를 낮추고 진입 장벽을 줄입니다. HubSpot의 "Marketing Grader"가 이 구조로 수백만 무료 사용자를 만들었습니다.

> **💡 아이디어 M3**: 랜딩 페이지에 "Quick Score" (3개 지표, 1분) 먼저 노출. Full Scan (15개, 10분)은 두 번째 단계. 진입 장벽 대폭 감소.

## M4. MODIFY — 수익 모델을 구독에서 성과 기반으로

## P1. VC 투자 실사(Due Diligence) 도구

**[민서]**: AI 스타트업에 투자하는 VC는 "이 회사의 AI 에이전트가 실제로 얼마나 잘 작동하는가"를 평가할 도구가 없습니다. HAchillesWorld가 이 용도로 쓰일 수 있습니다.

```
VC 실사 시나리오:
  AI 스타트업 투자 검토 → HAchillesWorld 진단 의뢰
  → 에이전트 World Model 점수 독립적 평가
  → 투자 의사결정의 기술 리스크 섹션에 반영
```

**[도현]**: 이 포지셔닝은 완전히 다른 GTM입니다. VC 10개사와 파트너십을 맺으면, 투자 대상 기업들이 자연스럽게 HAchillesWorld를 쓰게 됩니다. 강력한 B2B2B 채널입니다.

**[승규]**: VC가 실사에 쓰는 도구로 포지셔닝하면 "독립적 제3자 평가"의 신뢰가 필요합니다. 이해충돌이 없어야 합니다. HAchillesWorld가 동시에 컨설팅(개선 서비스)과 평가(독립 진단)를 하면 이해충돌 문제가 생깁니다. 역할 분리가 필요합니다.

> **💡 아이디어 P1**: "AI 스타트업 기술 실사" 서비스 패키지. VC/PE 파트너십으로 딜 플로우 확보. 단, 평가(Scan)와 컨설팅(Optimize) 역할은 엄격히 분리.

---

## P2. 정부·공공기관 AI 조달 기준

**[현아]**: 정부가 AI 시스템을 조달할 때 품질 기준이 없습니다. "이 AI 에이전트가 얼마나 믿을 수 있는가"를 평가하는 공식 기준이 없어서 가격 경쟁만 벌어집니다.

**[승규]**: 조달청·과학기술정보통신부와 협력해 "공공 AI 에이전트 품질 기준"으로 HAchillesWorld Levels × Laws를 제안할 수 있습니다. 국가 표준화(KS 규격)나 정부 가이드라인으로 채택되면 시장 표준이 됩니다.

**[민서]**: 공공 표준화는 오래 걸리지만 일단 채택되면 강력한 해자(moat)가 됩니다. 단기 수익보다 장기 시장 지위를 원한다면 정부 파트너십에 투자해야 합니다.

> **💡 아이디어 P2**: 과기부·조달청과 "공공 AI 에이전트 품질 기준" 공동 개발 제안. 규격 채택 시 시장 표준이 됨. 3~5년 장기 프로젝트로 접근.

---

## P3. AI 개발사(기업)가 아닌 AI 구매자(고객)에게 판매

**[도현]**: 지금 HAchillesWorld는 "AI 에이전트를 만드는 기업"에게 팔고 있습니다. 발상을 뒤집어, "AI 에이전트를 구매하는 기업"에게 팔면 어떨까요?

```
현재 고객: AI 에이전트 개발팀 → "내 에이전트를 개선하고 싶다"
대안 고객: AI 솔루션 구매 기업 → "외부에서 구매한 AI 에이전트가 믿을 수 있는지 검증하고 싶다"

예시:
  A기업이 B사의 AI 고객응대 에이전트를 도입하려 함
  → 계약 전 HAchillesWorld 독립 진단 의뢰
  → World Model 점수 700 이상이어야 계약
```

**[현아]**: 이것이 제조업에서 완제품 납품 전 품질검사와 같은 개념입니다. AI 에이전트 공급망에서 독립적 품질보증(QA) 기관이 됩니다.

**[승규]**: 이 역할을 하려면 "독립적 제3자"로서의 인증이 필요합니다. ISO 17025(시험기관 인증)나 유사한 국제 인정을 받으면 가능합니다.

> **💡 아이디어 P3**: "AI 에이전트 독립 품질보증(QA)" 서비스. 구매 기업이 납품 에이전트를 검증하는 용도. 장기적으로 독립적 시험기관 인증 획득 목표.

---

## P4. 대학교 AI 커리큘럼 도구

**[도현]**: AI 관련 학과가 폭발적으로 늘고 있습니다. HAchillesWorld의 Levels × Laws 프레임워크를 대학 커리큘럼에 제공하면:

```
교육 활용:
  학부: AI 에이전트 개론 실습 도구 (무료 교육 라이선스)
  대학원: World Model 연구 방법론 프레임워크
  산학협력: 실제 기업 에이전트 사례 연구
```

**[민서]**: 대학 교수들이 학생들에게 HAchillesWorld를 쓰게 가르치면, 그 학생들이 기업에 입사해서 HAchillesWorld를 도입합니다. 10년짜리 마케팅 전략입니다.

**[수진]**: 교육용 무료 버전 UI는 현재 다크 테마와 달리 밝고 설명적인 UX가 필요합니다. "학습 모드"를 별도로 설계해야 합니다.

> **💡 아이디어 P4**: 대학·연구기관 무료 교육 라이선스 프로그램. Tableau Academic Program처럼 미래 사용자를 양성. 책 출판과 연계하면 시너지 극대화.

# E — ELIMINATE (제거하라)

> **핵심 질문**: 빼면 오히려 좋아지는 것이 있는가?

---

## E1. "Operate" 탭을 메인 네비에서 제거

**[수진]**: Operate는 Scan → Report → Optimize를 완료한 사용자만 의미가 있습니다. 그런데 지금 메인 네비에 노출되어 있어서 처음 방문자가 클릭하면 맥락 없는 더미 대시보드를 봅니다.

**[도현]**: "기능이 많아 보인다" vs "진입 경험이 흐려진다" — 트레이드오프입니다. 처음 방문자 전환율이 더 중요하므로 Operate는 숨기고, Scan 완료 후에만 진입 가능하게 하는 것이 맞습니다.

> **💡 아이디어 E1**: Operate 링크를 메인 네비에서 제거. Scan Report 페이지 내 "Operate 시작하기" 버튼으로만 진입. 첫 방문자 경험 단순화.

---

## E2. 15개 지표 일괄 노출 제거

**[수진]**: 리포트에서 15개 지표를 한꺼번에 보여주면 대부분 사용자가 스크롤만 합니다. "정보는 많은데 무엇을 해야 할지 모른다"는 경험이 됩니다.

**[지훈]**: "3개 핵심 문제 + 순서 있는 해결 방법"이 훨씬 효과적입니다. 상위 3개 이슈를 빨간색으로 강조하고, 나머지 12개는 "상세 보기"로 숨깁니다.

```
제거 전: 15개 지표 모두 첫 화면에 노출
제거 후: Top 3 이슈 강조 → "지금 당장 이것을 고치세요"
         나머지는 "전체 리포트 보기" 클릭 시 표시
```

> **💡 아이디어 E2**: 리포트 첫 화면을 "Top 3 긴급 이슈"로 단순화. 전체 15개는 접혀있는 상세 섹션으로. 액션 가능성(actionability) 극대화.

---

## E3. 인증 프로그램 조기 노출 제거

**[민서]**: 신뢰가 쌓이기 전에 ₩500K짜리 인증을 홈 네비에 노출하면 오히려 역효과입니다. "돈 받는 게 목적인 사이트"로 보일 수 있습니다.

**[도현]**: Drift(Conversation AI 플랫폼)의 초기 전략이 이것입니다. 제품이 충분히 가치를 제공한 후에 유료를 꺼내라. Scan → 무료 가치 경험 → 신뢰 형성 → 그 다음에 인증.

> **💡 아이디어 E3**: 인증(cert.html)을 메인 네비에서 제거. Operate 완료 또는 일정 점수 달성 후 자연스럽게 "다음 단계로 공식 인증을 받으세요" 노출. 신뢰 선행, 수익 후행.

---

## E4. "얼리 액세스" 표현 제거 또는 명확화

**[수진]**: "얼리 액세스"는 두 가지 해석이 가능합니다: ① 나는 특별한 사람이고 이걸 먼저 쓴다 (긍정) ② 아직 완성되지 않은 제품이다 (부정). B2B 구매자는 ②로 읽습니다.

**[도현]**: 대안: "Partner Program" 또는 "Founder's Access". "얼리"를 빼고 특권 의식을 강조. "우리가 처음으로 선택한 파트너에게만 제공합니다"의 뉘앙스.

> **💡 아이디어 E4**: "Early Access" → "Founding Partner Access". 미완성 느낌 제거, 배타적 파트너십 느낌 강화.

---

## E5. 로그 업로드 방식 제거 → 코드 연동으로 전환

**[현아]**: 현재 스캔 방식은 "로그 파일을 올리세요"입니다. 그런데 기업이 로그 파일을 외부 서비스에 올리는 것은 보안 정책상 극히 어렵습니다.

**[지훈]**: 파일 업로드 방식을 아예 제거하고, `pip install hachillesworld` 후 코드 내 인스트루멘테이션으로만 데이터를 수집하는 방식이 보안 측면에서 훨씬 낫습니다. 데이터가 고객 서버를 벗어나지 않는 "로컬 실행" 옵션도 함께 제공합니다.

> **💡 아이디어 E5**: 파일 업로드 UI 제거 → SDK 설치 + 로컬 분석 방식으로 전환. "데이터는 고객 서버를 벗어나지 않습니다"가 가장 강력한 보안 메시지.

# R — REVERSE / REARRANGE (역전하라 / 재배열하라)

> **핵심 질문**: 순서를 뒤집거나 관계를 역전하면 새로운 가능성이 보이는가?

---

## R1. 도구를 먼저가 아니라, 책을 먼저

**[민서]**: 지금 전략: 도구(HAchillesWorld) 출시 → 책으로 신뢰 보완  
역전 전략: 책(《Agentic World Modeling 2027》) 출간 → 도구가 책의 실습 부록

```
역전 후 시퀀스:
  1. 책 출간 (신뢰 자산 확립)
  2. 책 독자 → 실습을 위해 HAchillesWorld 사용
  3. 사용 경험 → 유료 컨설팅·인증으로 전환
  4. 컨설팅 수익으로 SaaS 개발
  5. SaaS 출시
```

**[도현]**: 이것이 사실 가장 정석적인 "Thought Leadership → Product" 경로입니다. 클레이튼 크리스텐슨의 파괴적 혁신 이론이 컨설팅 수요를 만든 것처럼, 책이 제품의 수요를 만들어야 합니다.

**[수진]**: 웹사이트도 순서를 바꿔야 합니다. 지금은 "도구 → 책 언급"인데, "책 → 도구"로 재배열하면 홈 히어로 섹션에 책 표지가 먼저 나옵니다.

> **💡 아이디어 R1**: 책 출간을 최우선 마일스톤으로 설정. 웹사이트 히어로를 "책 + 도구"로 재배열. 지식 자산이 제품 판매의 선행 조건.

---

## R2. 기업이 도구를 사는 것이 아니라, 도구가 기업을 찾아가는 것

**[도현]**: 현재 모델: 기업이 HAchillesWorld.com에 방문 → 폼 작성  
역전 모델: HAchillesWorld가 GitHub/npm 패키지로 먼저 무료 배포 → 사용 데이터 축적 → 이후 유료 기능

```
역전 모델 (오픈소스 PLG):
  pip install hachillesworld  ← 완전 무료
  → 에이전트 진단 로컬 실행
  → "클라우드 대시보드로 팀과 공유" → 유료 전환
```

**[지훈]**: 이것이 Hugging Face, LangChain의 성장 방식입니다. 개발자가 먼저 쓰고, 나중에 팀이 유료로 전환합니다. HAchillesWorld CLI 오픈소스 공개가 가장 강력한 성장 레버가 될 수 있습니다.

**[승규]**: 오픈소스 코어(diagnostic)는 무료, 상업 기능(cloud dashboard, team sharing, enterprise support)은 유료인 Open Core 모델이 적합합니다. MongoDB, Elasticsearch가 이 방식으로 성장했습니다.

> **💡 아이디어 R2**: CLI 진단 코어를 MIT 라이선스로 오픈소스 공개. GitHub Stars를 신뢰 지표로 활용. 클라우드 기능만 유료화. 개발자 커뮤니티가 GTM의 엔진이 된다.

---

## R3. 컨설팅을 팔기 전에 진단을 먼저 무료로 주는 것을 역전

**[민서]**: 지금 전략: 무료 진단 → 유료 최적화  
역전 고민: 진단 자체가 가치라면, 진단도 유료로 팔 수 있을까?

```
유료 진단 모델:
  기업 에이전트 감사(Audit): ₩500K ~ ₩2M
  → 독립적 제3자 평가라는 가치
  → 이사회·투자자에게 제출 가능한 공식 감사 보고서
  → 컨설팅 연계 없는 순수 평가
```

**[현아]**: 보안 감사, 재무 감사를 떠올리면 자연스럽습니다. 독립적 외부 평가는 신뢰성 때문에 오히려 비싸야 합니다. "무료 진단"이 오히려 "가벼운 서비스"로 인식될 수 있습니다.

**[도현]**: 두 트랙을 분리합니다: ① 무료 Self-Scan (개발자·탐색 단계) ② 유료 Expert Audit (기업·공식 문서화). 같은 기술인데 포지셔닝이 다릅니다.

> **💡 아이디어 R3**: "Expert Audit" 유료 트랙 추가. ₩1M~₩5M 프리미엄 독립 감사 서비스. 무료 Self-Scan과 병행. 고객 세그먼트가 달라진다.

---

## R4. 에이전트가 실패한 후 진단이 아니라, 배포 전 예방적 진단으로 역전

**[지훈]**: 현재 포지셔닝: "에이전트가 실패한다 → HAchillesWorld로 진단"  
역전 포지셔닝: "에이전트를 배포하기 전 → HAchillesWorld로 검증"

```
사후 진단 (현재):
  에이전트 실패 → "왜 실패했는가?" → HAchillesWorld 진단
  
사전 검증 (역전):
  에이전트 개발 완료 → "배포해도 되는가?" → HAchillesWorld 통과 → 배포
  
차이:
  - 사후 진단: 문제가 이미 발생, 손실이 있음
  - 사전 검증: 문제 예방, 손실 없음
  → 사전 검증이 훨씬 더 높은 가치 제안
```

**[수진]**: 랜딩 카피도 바뀝니다. "에이전트가 왜 실패하는가"에서 "배포하기 전에 반드시 확인하세요"로. 후자가 훨씬 행동 유발적입니다.

**[민서]**: 이 포지셔닝은 CI/CD 통합과 직결됩니다. 아이디어 A3(Agent Deployment Gate)과 연결됩니다. "HAchillesWorld 통과 없이 프로덕션 배포 없다"는 파이프라인 표준이 되는 것입니다.

> **💡 아이디어 R4**: 포지셔닝을 사후 진단에서 사전 배포 게이팅으로 역전. 카피라이팅, CTA, 제품 진입점 모두 변경. "배포 전 체크리스트"가 새로운 유입 경로.

---

## R5. 고객이 HAchillesWorld를 배우는 것이 아니라, HAchillesWorld가 고객에게 적응

**[수진]**: 현재 모델: 사용자가 15개 지표, Levels × Laws 등을 배워야 함  
역전 모델: HAchillesWorld가 사용자의 언어로 자동 적응

```
역전 UX:
  초기 온보딩 질문: "당신은 어떤 역할인가요?"
  → 개발자   : CLI + 코드 중심 결과
  → 팀장     : 요약 대시보드 + 우선순위 목록
  → 임원     : ROI 수치 + 비교 벤치마크
  → 컨설턴트 : 원시 데이터 + API 접근
  
  같은 진단 결과, 다른 뷰
```

**[도현]**: Persona-based UI는 이미 Salesforce, HubSpot이 잘 하고 있습니다. "당신을 위한 맞춤 결과"는 강력한 첫 경험입니다.

> **💡 아이디어 R5**: 온보딩 페르소나 선택 → 맞춤형 결과 뷰 제공. 같은 데이터, 다른 언어. 개발자·임원·컨설턴트 모두를 만족시키는 적응형 UI.

# 최종 메타 통찰 — SCAMPER 토론에서 공통으로 나온 주제

**[지훈]**: 기술적으로 가장 일관된 메시지는 "**단순화**"입니다. 15개 지표, MCTS, Levels×Laws — 다 좋지만 너무 복잡합니다. 복잡성을 뒤에 숨기고 단순한 숫자 하나를 앞에 내세워야 합니다.

**[수진]**: UX에서 나온 공통 주제는 "**순서**"입니다. 지금 너무 많은 것을 동시에 보여줍니다. 무엇을 먼저, 무엇을 나중에 보여줄지 결정하면 전환율이 달라집니다.

**[도현]**: 마케팅 공통 주제는 "**신뢰의 순서**"입니다. 지식(책·블로그) → 작은 경험(무료 Quick Scan) → 큰 경험(Full Scan) → 관계(컨설팅) → 자격(인증). 이 순서를 따르면 됩니다.

**[현아]**: 현장에서 나온 공통 주제는 "**첫 번째 마찰 제거**"입니다. 지금 첫 번째 마찰은 "로그 파일이 뭐지?"입니다. 이걸 없애면 나머지는 따라옵니다.

**[승규]**: 법적 공통 주제는 "**수익 발생 전 기반 구축**"입니다. 상표, 수료 고지, 데이터 정책 — 지금 0원이고 작업 시간도 크지 않은데 나중에 매우 크게 작용합니다.

**[민서]**: 전략적 공통 주제는 "**지금 가진 것을 먼저 활용하라**"입니다. 박성훈이라는 저자, Levels×Laws라는 프레임워크, 3편의 좋은 블로그 글 — 이것들이 이미 충분한 시작점입니다. 새로운 것을 만들기 전에 있는 것을 먼저 제대로 쓰십시오.
