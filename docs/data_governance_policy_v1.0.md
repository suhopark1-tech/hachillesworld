## 1. 목적 및 적용 범위

### 1.1 목적

이 규정은 HAchillesWorld(이하 "회사")가 운영하는 AI 에이전트 진단 플랫폼에서 수집·처리·활용되는 고객 데이터의 관리 원칙과 절차를 정함으로써:

1. 고객의 개인정보 및 에이전트 진단 데이터를 안전하게 보호한다.
2. 관련 법령(개인정보보호법, GDPR, EU AI Act)을 성실히 준수한다.
3. 수집한 데이터를 목적에 맞게 활용하여 플랫폼 품질을 지속적으로 향상한다.
4. 고객의 신뢰를 유지하고 데이터 기반 성장의 지속 가능성을 확보한다.

### 1.2 적용 범위

이 규정은 다음에 모두 적용된다.

```
적용 대상 데이터:
  · 플랫폼 가입 및 이용 과정에서 수집되는 고객 식별 정보
  · AI 에이전트 스캔·진단 과정에서 수집되는 EpisodeRecord
  · 생성되는 DiagnosticReport, HASTimeSeries, DriftLog
  · 플랫폼 이용 행동 데이터(CustomerEvent)
  · 감사 이벤트(AuditEvent)
  · 위로부터 파생·집계된 모든 분석 데이터

적용 대상 인원:
  · 회사의 모든 임직원 및 계약직
  · 데이터 처리 업무를 위탁받은 외부 업체 담당자
  · 회사 시스템에 접근 권한을 부여받은 모든 자

적용 제외:
  · 공개 벤치마크 데이터 (익명화 완료, 재식별 불가 확인된 집계값)
  · 회사 내부 운영 데이터 (인사, 재무 등)
```

### 1.3 최고 책임자 및 담당자

| 역할 | 담당자 | 책임 |
|------|--------|------|
| 개인정보 보호책임자 (CPO) | 박성훈 (대표) | 규정 최종 승인, 법적 책임 |
| 데이터 관리자 | 개발팀 리드 | 기술적 보호조치 이행, DDS 설계 관리 |
| 데이터 품질 담당 | 운영팀 | DQI 측정, 이상 감지 대응 |
| 고객 권리 대응 담당 | 고객지원팀 | 열람·삭제 요청 처리 |

> 조직 확장 시 별도 RACI 매트릭스로 세분화한다.

## 2. 용어 정의

이 규정에서 사용하는 주요 용어의 정의는 다음과 같다.

| 용어 | 정의 |
|------|------|
| **고객** | 플랫폼에 가입하여 서비스를 이용하는 개인 또는 법인 |
| **정보주체** | 처리되는 개인정보의 당사자인 자연인 |
| **에이전트** | 고객이 플랫폼에 등록하여 진단 받는 AI 에이전트 |
| **EpisodeRecord** | 에이전트 1회 실행의 원시 수집 데이터 단위 (DDS §3.6) |
| **DiagnosticReport** | 스캔 세션으로부터 생성된 HAS 점수 및 15개 지표 진단 결과 (DDS §3.5) |
| **HAS 점수** | HAchilles Agent Score — 에이전트 World Model 품질 종합 점수 (0~100) |
| **CustomerConsent** | 고객의 데이터 활용 동의 정보 (DDS §3.10) |
| **Level 0~3** | 데이터 보안 등급 분류 (DDS §1.3, 본 규정 §3) |
| **DQI** | Data Quality Index — 데이터 품질 종합 지표 (DDS §10.2) |
| **익명화** | 재식별이 불가능하도록 개인식별 정보를 완전히 제거하는 처리 |
| **가명화** | 식별 가능성을 낮추되 추가 정보로 복원 가능한 처리 |
| **K-익명성** | 집계 데이터 내 최소 K개 이상의 레코드가 동일한 특성을 가지도록 보장 |
| **BenchmarkBuilder** | 동의 고객 데이터를 집계하여 도메인별 HAS 벤치마크를 생성하는 내부 시스템 |
| **Cold Store** | S3 기반 원시 데이터 장기 보관 스토리지 (DDS §7.1 Tier 3) |
| **CPO** | 개인정보 보호책임자 (Chief Privacy Officer) |

## 3. 데이터 분류 및 보안 등급

### 3.1 4단계 보안 등급 체계

모든 데이터는 수집 시점에 다음 등급 중 하나로 분류된다.

```
┌─────────────────────────────────────────────────────────────────┐
│  Level 0 — 고객 식별 원시 데이터 (최고 보안)                      │
│                                                                   │
│  데이터: 이메일, 회사명, API 키, 에이전트 이름                     │
│  저장:   pgcrypto 암호화 (AES-256) + bcrypt 해시                  │
│  접근:   CPO 및 시스템 자동화만 허용, 직접 조회 금지               │
│  활용:   인증 수단으로만 사용, 마케팅·분석 활용 금지               │
│  보관:   계약 유지 기간 + 법적 의무 기간                           │
└─────────────────────────────────────────────────────────────────┘
              ↓ 동의 및 목적 기준 하향 이동 가능
┌─────────────────────────────────────────────────────────────────┐
│  Level 1 — 고객별 진단 결과 (고보안)                              │
│                                                                   │
│  데이터: DiagnosticReport, HASTimeSeries, DriftLog,              │
│          EpisodeRecord, HarnessRule, AuditEvent                  │
│  저장:   PostgreSQL + TimescaleDB (고객별 논리 격리)              │
│  접근:   해당 고객 본인 + 시스템 서비스 계정만 허용               │
│  활용:   해당 고객에 대한 서비스 제공 목적에 한함                  │
│  보관:   플랜별 기간 (§6.2 참조) + 고객 동의 설정 반영            │
└─────────────────────────────────────────────────────────────────┘
              ↓ anonymized_benchmark = TRUE 동의 시
┌─────────────────────────────────────────────────────────────────┐
│  Level 2 — 익명화 집계 데이터 (중보안)                            │
│                                                                   │
│  데이터: benchmark_snapshots, metric_distributions,              │
│          failure_pattern_stats, improvement_stats                │
│  저장:   Benchmark Store (별도 스키마, 원본과 물리 분리)           │
│  접근:   내부 분석팀, 제품팀 (개인 식별 불가 검증 후)             │
│  활용:   제품 개선, 내부 인사이트, 웨비나 자료                     │
│  보관:   무기한 (재식별 불가 확인 조건)                            │
└─────────────────────────────────────────────────────────────────┘
              ↓ public_case_study = TRUE 동의 시
┌─────────────────────────────────────────────────────────────────┐
│  Level 3 — 공개 벤치마크 데이터 (공개)                            │
│                                                                   │
│  데이터: GET /v1/public/benchmark API 응답,                       │
│          웹사이트 /benchmark 페이지 데이터                         │
│  저장:   CDN 캐시 허용                                            │
│  접근:   인터넷 공개                                              │
│  활용:   마케팅, 신규 고객 유치, 학술 연구 참고                    │
│  보관:   공개 기간 동안                                            │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 등급별 필수 보호조치

| 보호조치 | Level 0 | Level 1 | Level 2 | Level 3 |
|---------|:-------:|:-------:|:-------:|:-------:|
| 암호화 저장 | 필수 | 필수 | 불필요 | 불필요 |
| 전송 시 TLS 1.3 | 필수 | 필수 | 필수 | 필수 |
| 접근 로그 기록 | 필수 | 필수 | 권고 | 불필요 |
| 다중 인증 (MFA) | 필수 | 필수 | 권고 | 불필요 |
| 정기 접근 권한 검토 | 분기별 | 반기별 | 연간 | 해당없음 |
| 백업 암호화 | 필수 | 필수 | 권고 | 불필요 |
| 삭제 요청 이행 | 30일 내 | 30일 내 | 이행불가(재식별불가) | 이행불가 |

## 4. 수집 원칙 및 허용 범위

### 4.1 수집의 7대 원칙

```
원칙 1. 목적 명확성
  수집 전 처리 목적을 명확히 고객에게 고지한다.
  수집 후 당초 목적 외 활용 시 별도 동의를 획득한다.

원칙 2. 최소 수집
  서비스 제공에 필요한 최소한의 데이터만 수집한다.
  에이전트의 실제 업무 콘텐츠(원문 텍스트, 고객 데이터)는
  수집하지 않는다. 수치·메타데이터만 수집한다.

원칙 3. 동의 우선
  수집 전 반드시 동의를 획득한다.
  동의 없이 수집한 데이터는 즉시 삭제한다.

원칙 4. 정확성
  수집된 데이터가 실제를 반영하도록 검증 절차를 둔다.
  이상 데이터(DQI 기준 이탈값)는 플래그 처리 후 진단에서 제외한다.

원칙 5. 불변성 (Immutability)
  수집된 DiagnosticReport는 수정하지 않는다.
  보정이 필요한 경우 새 레코드를 append하고 기존 레코드를 유지한다.

원칙 6. 안전성
  수집 시점부터 암호화·접근 통제를 적용한다.

원칙 7. 투명성
  고객이 언제든 자신의 수집 데이터 현황을 조회할 수 있어야 한다.
```

### 4.2 수집 허용 데이터 목록

| 데이터 항목 | 수집 근거 | 비고 |
|------------|---------|------|
| 이메일 주소 | 서비스 계약 | 암호화 저장 |
| 회사명 | 서비스 계약 | 암호화 저장, 선택 |
| API 키 | 서비스 계약 | 해시만 저장 |
| 에이전트 도메인·태그 | 서비스 계약 | 진단 분류 목적 |
| EpisodeRecord 수치 필드 | 서비스 계약 | 15개 지표 산출용 |
| DiagnosticReport | 서비스 제공 | 고객 조회 가능 |
| HASTimeSeries | 서비스 제공 | 대시보드 표시 |
| DriftLog (수치만) | 서비스 제공 | 원시 상태 페이로드 제외 |
| CustomerEvent (행동) | 서비스 개선 동의 | product_improvement=TRUE |
| AuditEvent | 법적 의무·보안 | 법령 근거, 동의 불필요 |

### 4.3 수집 금지 데이터

다음 데이터는 어떠한 경우에도 수집하지 않는다.

```
절대 수집 금지:
  · 에이전트가 처리하는 최종 사용자의 개인정보
  · predicted_next_state / actual_next_state의 원문 텍스트
    (JSON 구조·수치만 허용, 4KB 크기 제한 적용)

  · 생체정보, 민감정보(건강, 신앙, 정치, 성적 지향 등)
  · 만 14세 미만 아동의 개인정보
  · 수집 목적이 불분명한 모든 데이터
```

## 5. 고객 동의 관리

### 5.1 동의 항목 및 분류

`CustomerConsent` 테이블의 4개 항목은 법적 근거에 따라 다음과 같이 분류된다.

| 동의 항목 | 법적 근거 | 기본값 | 거부 시 영향 |
|---------|---------|:-----:|------------|
| `anonymized_benchmark` | 정보주체 동의 | FALSE | 벤치마크 집계에 미포함 (서비스 이용 가능) |
| `public_case_study` | 정보주체 동의 | FALSE | 케이스스터디 미게재 (서비스 이용 가능) |
| `product_improvement` | 정보주체 동의 | FALSE | CustomerEvent 수집 안 함 |
| `marketing_contact` | 정보주체 동의 | FALSE | 마케팅 이메일 미발송 |

> **서비스 제공 필수 동의**: 이메일·API 키 수집은 서비스 계약에 근거하며 별도 동의 항목이 아니다.

### 5.2 동의 획득 절차

```
Step 1. 가입 온보딩 화면
  표시 내용:
    · 개인정보처리방침 전문 링크
    · 수집 항목·목적·보관 기간 요약 (3줄 이내)
  필수 확인:
    □ [필수] 개인정보 수집·이용에 동의합니다 (이메일·API키)
    □ [선택] 서비스 개선을 위한 사용 패턴 분석에 동의합니다
    □ [선택] 익명화된 벤치마크 집계에 기여하는 것에 동의합니다
    □ [선택] HAchilles Weekly 뉴스레터 수신에 동의합니다
  처리:
    PUT /v1/consent 호출 → CustomerConsent 레코드 생성
    동의 시각·버전 기록 (consented_at, consent_version)

Step 2. 첫 스캔 완료 후 팝업 (anonymized_benchmark)
  · 미동의 고객에게만 1회 표시
  · "이 결과를 익명 벤치마크에 기여하시겠어요?"
  · "언제든 설정에서 변경 가능" 안내

Step 3. 케이스스터디 제안 (public_case_study)
  · HAS 점수 유의미 개선 고객에게 개별 이메일·인터뷰로 별도 제안
  · 제안 전 product_improvement 또는 marketing_contact 동의 확인 필수
```

### 5.3 동의 철회 처리

```
고객의 동의 철회 요청 수신 시:
  1. PUT /v1/consent 또는 설정 페이지에서 즉시 반영
  2. marketing_contact 철회 → 발송 예정 이메일 큐에서 즉시 제거 (1시간 내)
  3. anonymized_benchmark 철회 → 다음 BenchmarkBuilder 실행 시 제외
     (이미 집계된 Level 2~3 데이터는 재식별 불가능하므로 제거 불가 — 고객에게 고지)

  5. 철회 처리 완료 시 확인 이메일 발송 (24시간 내)
```

## 6. 데이터 저장 및 보관 정책

### 6.1 스토리지 계층별 관리 원칙

DDS §7.1의 3-Tier 스토리지에 따라 다음 원칙을 적용한다.

```
Hot Store (PostgreSQL + TimescaleDB):
  · 보관 기간: 최근 90일 (플랜별 상이)
  · 암호화: 저장 데이터 AES-256, 전송 TLS 1.3
  · 백업: 매일 자동 백업, 7일 보관
  · 접근: 서비스 계정 + 권한 있는 임직원만

Warm Store (TimescaleDB 자동 압축):
  · 보관 기간: 90일 ~ 1년
  · 압축: TimescaleDB 청크 자동 압축 적용
  · 접근: 읽기 전용 레플리카를 통해서만

Cold Store (S3):
  · 보관 기간: 1년 이상 (고객 동의 설정 따름)
  · 암호화: S3 Server-Side Encryption (SSE-S3 또는 SSE-KMS)
  · 버킷 정책: 퍼블릭 접근 완전 차단
  · 접근 로그: S3 Access Log 활성화
```

## 7. 접근 통제 및 권한 관리

### 7.1 접근 권한 원칙

```
최소 권한 원칙 (Principle of Least Privilege):
  · 각 임직원은 업무에 필요한 최소한의 데이터에만 접근 가능
  · 직무 변경 시 즉시 권한 재조정

Need-to-Know 원칙:
  · Level 0 데이터는 CPO와 자동화 시스템만 접근 가능
  · 마케팅팀은 Level 2~3 집계 데이터만 접근 가능
  · 고객지원팀은 해당 고객의 Level 1 데이터에 한해 접근 가능

직무 분리 원칙:
  · 데이터 수집 담당자 ≠ 데이터 분석 담당자
  · 삭제 요청 처리 담당자 ≠ 해당 데이터 개발 담당자
```

### 7.2 권한 등급별 접근 범위

| 역할 | Level 0 | Level 1 | Level 2 | Level 3 | AuditEvent |
|------|:-------:|:-------:|:-------:|:-------:|:----------:|
| 시스템 서비스 계정 | 읽기 | 읽기/쓰기 | 읽기/쓰기 | 읽기/쓰기 | 쓰기 |
| CPO (박성훈) | 읽기 | 읽기 | 읽기 | 읽기/쓰기 | 읽기 |
| 개발팀 리드 | 구조만 | 구조만 | 읽기 | 읽기/쓰기 | 읽기 |
| 개발팀 일반 | 없음 | 없음 | 읽기 | 읽기 | 없음 |
| 마케팅팀 | 없음 | 없음 | 읽기 | 읽기/쓰기 | 없음 |
| 고객지원팀 | 없음 | 해당 고객만 | 없음 | 없음 | 없음 |
| 외부 위탁업체 | 없음 | 없음 | 없음 | 없음 | 없음 |

### 7.3 인증 및 접근 통제 기술 요건

```
데이터베이스 접근:
  · 다중 인증(MFA) 필수 (Level 0~1 접근 시)
  · IP 화이트리스트 적용 (사무실 IP + VPN)
  · 세션 타임아웃: 30분 비활동 시 자동 잠금

  · Bearer Token (API 키 해시 검증)

관리자 콘솔:
  · SSO + MFA 필수
  · 모든 조회·수정 작업 로그 기록
  · 프로덕션 DB 직접 접근 금지 (읽기 전용 레플리카 활용)
```

## 8. 데이터 품질 관리

### 8.1 DQI(Data Quality Index) 측정 기준

DDS §10.2에 정의된 5개 항목을 매일 자동 측정한다.

```
DQI 측정 항목 및 목표:

  Completeness (완전성)
    측정: 필수 필드 채워진 에피소드 수 / 전체 에피소드 수
    목표: > 0.95 (95% 이상)
    위반: DQI 경보 → 해당 세션 진단 결과에 "데이터 품질 경고" 표시

  Validity (유효성)
    측정: 범위 검사 통과 에피소드 수 / 전체 에피소드 수
    목표: > 0.99 (99% 이상)
    위반: 이상 에피소드 플래그 처리 + 해당 고객 알림

  Timeliness (적시성)
    측정: occurred_at → recorded_at 지연 (분)
    목표: < 5분 (중앙값 기준)
    위반: 배치 전송 설정 점검 권고

  Consistency (일관성)
    측정: 동일 에이전트 내 이상값 비율
    목표: < 0.01 (1% 미만)
    위반: 고객 SDK 설정 점검 권고

  Uniqueness (고유성)
    측정: episode_id 중복 비율
    목표: 0% (완전 고유)
    위반: 중복 레코드 즉시 플래그 처리

DQI 종합 점수 = (Completeness × 0.3) + (Validity × 0.3)
                + (Timeliness × 0.2) + (Consistency × 0.1)
                + (Uniqueness × 0.1)

DQI ≥ 0.85: 정상
0.80 ≤ DQI < 0.85: 주의 (담당자 알림)
DQI < 0.80: 경보 (진단 결과에 경고 표시 + CPO 보고)
```

### 8.2 이상 탐지 규칙

```
자동 이상 탐지 (IngestValidator 적용):

  Rule 1. 단일 세션 에피소드 과다
    조건: episode_count > 10,000
    처리: 초과분 수신 거부 + 고객 알림

  Rule 2. 완벽한 성공률 의심
    조건: 세션 내 goal_achieved = TRUE 비율 = 100%
    처리: AnomalyFlag 기록 + 진단 결과에 "데이터 편향 주의" 표시

  Rule 3. 필드값 범위 위반
    조건: confidence ∉ [0,1], planning_depth > 100, llm_tokens > 1,000,000 등
    처리: 해당 에피소드 rejection + 유효 에피소드만으로 진단

  Rule 4. 에피소드 지연 과다
    조건: recorded_at - occurred_at > 24시간
    처리: Timeliness 경고 + 고객 SDK 재설정 권고

  Rule 5. 동일 episode_id 중복
    조건: INSERT 시 UNIQUE 제약 위반
    처리: 중복 레코드 무시 + 멱등성 보장
```

### 8.3 벤치마크 데이터 신뢰성 기준

```
BenchmarkBuilder 실행 전 검증 (매일 00:00 KST):

  K-익명성 검증:
    조건: 도메인+레벨 조합별 에이전트 수 ≥ 5  (K=5 임계값 유지)
    미달 시: 해당 조합 집계 생략, 전일 스냅샷 유지

  재식별 방지 — 공개 API 노출 방식 (HAW-LGL-002):
    내부 집계: agent_count 정수값 보관 (BenchmarkBuilder → benchmark_snapshots)
    외부 노출: agent_count_range 구간 문자열로 변환 후 공개
               5~9개 / 10~29개 / 30~99개 / 100개 이상
    근거: 구간 표시는 K=5 임계값 + 불확실성 증가를 결합하여
          K=10 정수 노출과 동등한 재식별 보호 수준 달성
          (HAW-LGL-002 §4.1 정량 분석 참조)
    추가: snapshot_period를 월 단위로 뭉쳐 제공 (정확한 스냅샷 날짜 미노출)

  L-다양성 검증:
    조건: 집계 그룹 내 HAS 점수 stddev > 5.0
    미달 시: 공개 API에서 해당 조합 404 반환

  버전 일관성:
    조건: has_score_version 동일한 데이터만 집계
    혼재 시: 최신 버전 기준으로 집계, 구버전 데이터 제외

  시간 기준:
    최신 30일 데이터만 집계 (오래된 데이터 자동 제외)
```

### 9.1 익명화 처리 원칙


### 9.2 익명화 처리 절차 (DataAnonymizer)

```python
# DDS §8.2를 내부 규정으로 명문화

익명화 처리 흐름:
  Step 1. anonymized_benchmark = TRUE 동의 고객 데이터 조회

          agent_id → NULL
          agent_name → NULL
  Step 3. 수치 데이터 Laplace 노이즈 처리:

  Step 4. 원시 페이로드 제거:
          predicted_next_state → 삭제

  Step 5. 집계 단위로 변환 (개별 레코드 미보존)
  Step 6. benchmark_snapshots INSERT (별도 DB)
  Step 7. 원본 Level 1 데이터는 그대로 유지 (고객 서비스용)
```

## 10. 제3자 제공 및 위탁 처리

### 10.1 제3자 제공 원칙

```
고객 데이터(Level 0~1)의 제3자 제공은 다음 경우에만 허용된다:
  1. 정보주체(고객)의 명시적 동의
  2. 법령의 규정에 의한 경우 (수사기관 요청 등)
  3. 계약 이행에 필수적인 경우 (결제 대행사 등)

Level 2~3 익명화 집계 데이터:
  · 재식별 불가 확인 후 제한 없이 제공 가능
  · 단, 수령 측이 재식별을 시도하지 않는다는 계약 조항 필수
```

### 10.2 위탁 처리 관리

현재 위탁 처리 현황 및 관리 기준:

| 수탁사 | 처리 내용 | 데이터 등급 | 계약 유효 기간 |
|--------|---------|:----------:|:------------:|
| AWS (Amazon) | S3 Cold Store, RDS | Level 0~1 | 연간 갱신 |
| (결제 대행사) | 구독 결제 처리 | Level 0 (이메일만) | 연간 갱신 |
| (이메일 발송) | HAchilles Weekly 발송 | Level 0 (이메일만) | 연간 갱신 |

```
위탁 계약 필수 조항:
  ① 목적 외 처리 금지
  ② 재위탁 금지 (사전 승인 없이)
  ③ 보안 요건 준수 의무
  ④ 위탁 종료 시 데이터 즉시 반환 또는 파기
  ⑤ 보안 사고 발생 시 24시간 내 통지 의무
  ⑥ 연 1회 이상 보안 점검 허용 의무
```

## 11. 정보주체의 권리 보장

### 11.1 보장 권리 목록

| 권리 | 내용 | 이행 기한 | 처리 방법 |
|------|------|:--------:|---------|
| 열람권 | 수집된 자신의 데이터 확인 | 10일 내 | GET /v1/data/me |
| 정정권 | 부정확한 데이터 수정 요청 | 10일 내 | 고객지원팀 이메일 |
| 삭제권 | 데이터 삭제 요청 (Right to Erasure) | **30일 내** | DELETE /v1/data/me |
| 처리 정지권 | 특정 목적 처리 중단 요청 | 즉시 | PUT /v1/consent |
| 이동권 | 자신의 데이터를 다른 서비스로 이전 | 20일 내 | JSON 내보내기 제공 |
| 자동화 거부권 | 자동화 진단·추천에만 의존하지 않을 권리 | 즉시 | 설정에서 선택 가능 |

### 11.2 삭제 요청 처리 절차 (DataErasureService)

```
DDS §8.3의 DataErasureService 실행 절차:

  요청 접수 (DELETE /v1/data/me or 이메일):
    → ErasureRequest 생성, 30일 타이머 시작

  Day 1~3: 신원 확인
    → 이메일 인증 또는 API 키 확인
    → 대리인 요청 시 위임장 확인

  Day 4~10: Cold Store 원시 로그 삭제
    → S3에서 해당 customer_id 버킷 파일 삭제
    → 삭제 완료 SHA-256 해시 기록 (삭제 증명)

  Day 11~20: Hot/Warm Store 삭제
    → episode_records, diagnostic_reports 삭제
    → has_timeseries, drift_logs, customer_events 삭제
    → harness_rules 삭제

  Day 21~25: 고객 계정 소프트 삭제
    → customers.deleted_at = NOW()
    → customers.email = "deleted_{customer_id}" (원본 덮어쓰기)
    → customers.company_name = NULL

  Day 26~30: 검증 및 완료 통보
    → 전체 테이블에서 customer_id 잔여 레코드 0건 확인
    → 삭제 완료 이메일 발송
    → ErasureReport 생성 (내부 감사용)

  삭제되지 않는 데이터 (고객에게 명시적 고지):
    → AuditEvent: 법적 보관 의무 (3년) — 법령 근거
    → 이미 집계된 Level 2~3 익명화 데이터 — 재식별 불가능
```

### 11.3 권리 행사 거부 사유

다음 경우 권리 행사를 제한할 수 있다 (고객에게 이유 명시 필수).
