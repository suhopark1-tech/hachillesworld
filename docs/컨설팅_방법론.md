## 1. 컨설팅 철학

### 1.1 핵심 신념

> **"에이전트가 실패할 때, 진짜 문제는 코드가 아니라 World Model이다."**

HAchillesWorld 컨설팅은 세 가지 신념을 기반으로 한다.

**신념 1: 진단 우선**  
최적화보다 진단이 먼저다. 문제를 정확히 이해하지 않은 채 해결책을 제시하는 컨설팅은 도박이다. 모든 인게이지먼트는 반드시 체계적 진단으로 시작한다.

**신념 2: 증거 기반**  
직관이 아닌 데이터로 말한다. HAchillesWorld Scan의 15개 지표, 운영 로그, 실제 실패 사례가 모든 권고의 근거가 된다. "저는 이렇게 생각합니다"가 아니라 "데이터는 이렇게 말합니다"로 말한다.

**신념 3: 고객 역량 강화**  
컨설턴트가 없어도 고객이 스스로 운영할 수 있어야 한다. 모든 인게이지먼트의 최종 목표는 고객 팀이 HAchillesWorld를 내재화하는 것이다. 컨설턴트 의존성을 만드는 것은 실패다.

### 1.2 방법론 기반 이론

HAchillesWorld 컨설팅 방법론은 《Agentic World Modeling 2027》의 세 가지 핵심 이론 위에 구축된다.

```
이론 1: Levels × Laws 프레임워크
  → 고객의 에이전트를 2차원 좌표계에 정확히 위치시킨다
  → "당신의 에이전트는 L2.3 × Digital Laws입니다"

이론 2: Harness Engineering
  → Mitchell Hashimoto: "에이전트가 같은 실수를 반복하지 않도록"
  → 모든 실패를 하네스 규칙으로 전환하는 피드백 루프 설계

이론 3: DEOR 자율 루프
  → Design → Execute → Observe → Reflect
  → 컨설팅 자체도 DEOR 루프로 운영한다
```

### 1.3 컨설팅 원칙 10가지

1. **솔직함**: 고객이 듣고 싶은 말이 아니라 필요한 말을 한다
2. **구체성**: 추상적 조언이 아닌 실행 가능한 태스크로 전달한다
3. **측정 가능성**: 모든 목표는 수치로 표현한다
4. **현실성**: 6주 안에 실현 가능한 것과 6개월이 필요한 것을 구분한다
5. **안전 우선**: ROI보다 안전 설계가 먼저다
6. **고객 맥락 우선**: 책의 이론을 고객 도메인에 맞게 변환한다
7. **문서화**: 모든 결정과 근거를 문서로 남긴다
8. **지식 이전**: 고객 팀이 배우고 자립할 수 있도록 한다
9. **장기 관점**: 단기 성과보다 지속 가능한 역량 구축을 우선한다
10. **투명성**: 우리가 모르는 것은 모른다고 말한다

### 4.3 World Model 품질 심층 분석

#### 4.3.1 Simulation Drift 패턴 분석

```python
# 드리프트 발생 시간대·상황별 패턴 분석
from hachillesworld.operate.monitor import DriftMonitor
from collections import defaultdict
import numpy as np

def analyze_drift_patterns(logs: list[dict]) -> dict:
    """드리프트가 언제, 어떤 상황에서 발생하는지 패턴 분석."""
    drift_by_hour     = defaultdict(list)
    drift_by_action   = defaultdict(list)
    drift_by_context  = defaultdict(list)

    for i, event in enumerate(logs):
        if event.get("event_type") != "reflect":
            continue
        payload  = event.get("payload", {})
        drift    = payload.get("prediction_error", 0.0)
        hour     = int(event.get("timestamp", 0) % 86400 // 3600)
        action   = logs[i-2].get("payload", {}).get("action", "unknown") if i >= 2 else "unknown"

        drift_by_hour[hour].append(drift)
        drift_by_action[action].append(drift)

    return {
        "peak_drift_hours": sorted(
            drift_by_hour.items(),
            key=lambda x: np.mean(x[1]), reverse=True
        )[:3],
        "high_drift_actions": sorted(
            drift_by_action.items(),
            key=lambda x: np.mean(x[1]), reverse=True
        )[:5],
        "overall_drift_rate": sum(
            1 for e in logs
            if e.get("event_type") == "reflect"
            and e.get("payload", {}).get("recalibrated")
        ) / max(1, sum(1 for e in logs if e.get("event_type") == "reflect")),
    }
```

#### 4.3.2 Calibration 오차 심층 진단

```python
def diagnose_calibration_failure(logs: list[dict]) -> dict:
    """
    캘리브레이션 실패의 종류를 구분한다.
    
    Type 1: 과신(Overconfidence) — 높은 확신, 낮은 실제 정확도
    Type 2: 과소신(Underconfidence) — 낮은 확신, 높은 실제 정확도
    Type 3: 불안정 캘리브레이션 — 확신과 정확도 모두 불규칙
    """
    plan_events    = [e for e in logs if e.get("event_type") == "plan"]
    observe_events = [e for e in logs if e.get("event_type") == "observe"]

    if not plan_events or not observe_events:
        return {"type": "insufficient_data"}

    confidences = [e.get("payload", {}).get("confidence", 0.5) for e in plan_events]
    accuracies  = [
        1.0 if e.get("payload", {}).get("goal_achieved") else 0.0
        for e in observe_events
    ]

    avg_conf = sum(confidences) / len(confidences)
    avg_acc  = sum(accuracies[:len(confidences)]) / len(accuracies[:len(confidences)])

    if avg_conf > avg_acc + 0.15:
        cal_type = "overconfidence"
        severity = "critical" if avg_conf - avg_acc > 0.30 else "warning"
    elif avg_acc > avg_conf + 0.15:
        cal_type = "underconfidence"
        severity = "warning"
    else:
        cal_type = "stable"
        severity = "ok"

    return {
        "type":           cal_type,
        "severity":       severity,
        "avg_confidence": round(avg_conf, 3),
        "avg_accuracy":   round(avg_acc, 3),
        "gap":            round(avg_conf - avg_acc, 3),
        "recommendation": {
            "overconfidence":  "Temperature 낮추기, Uncertainty 표현 추가, 앙상블 방식 도입",
            "underconfidence": "계획 다양성 증가, 탐색 강화, 목표 재정의",
            "stable":          "현 상태 유지, 장기 추이 모니터링",
        }.get(cal_type, ""),
    }
```

**Laws 부적합 징후 체크리스트**:

```

□ Social Laws 시스템이 다른 에이전트를 신뢰 없이 협력 → 충돌, 교착
□ Scientific Laws 시스템이 인과 관계 없이 상관관계로 결론 → 재현 불가 결과
```

### 5.5 ROI 계산 방법론

```python
def calculate_roi(
    current_report,
    target_score: float,
    monthly_budget_usd: float,
    consulting_cost_usd: float,
    implementation_cost_usd: float,
) -> dict:
    """
    컨설팅 인게이지먼트의 예상 ROI를 계산한다.
    
    절감 원천:
    1. 비용 최적화 (라우팅, 캐싱)
    2. 실패 감소 (재작업 비용)
    3. HITL 감소 (운영 인건비)
    4. 속도 향상 (에이전트 처리 효율)
    """
    # 1. 비용 최적화 절감 추정
    cost_metric = next(
        (m for m in current_report.operational_health.metrics
         if m.name == "Cost Efficiency"), None
    )
    current_efficiency = cost_metric.value if cost_metric else 1.0
    target_efficiency  = 0.70  # 목표: 예산의 70% 사용
    cost_saving_ratio  = max(0, current_efficiency - target_efficiency)
    annual_cost_saving = monthly_budget_usd * 12 * cost_saving_ratio

    # 2. 실패 감소 절감 추정
    # 에이전트 실패 1건당 평균 복구 비용: $500 (가정)
    current_drift_rate = next(
        (m.value for m in current_report.world_model_quality.metrics
         if "Drift" in m.name), 0.15
    )
    target_drift_rate  = 0.05
    drift_reduction    = max(0, current_drift_rate - target_drift_rate)
    monthly_failures   = drift_reduction * 100  # 월 100 에피소드 가정
    failure_saving     = monthly_failures * 500 * 12  # 연간

    # 3. HITL 절감 추정
    # HITL 1건당 엔지니어 15분 소요, 시급 $100
    hitl_metric = next(
        (m for m in current_report.operational_health.metrics
         if "HITL" in m.name), None
    )
    current_hitl = hitl_metric.value if hitl_metric else 0.10
    target_hitl  = 0.05
    hitl_reduction_rate = max(0, current_hitl - target_hitl)
    monthly_tasks = 1000  # 월 1,000 태스크 가정
    hitl_saving   = hitl_reduction_rate * monthly_tasks * (15/60 * 100) * 12

    # 총 절감액
    total_annual_saving = annual_cost_saving + failure_saving + hitl_saving
    total_investment    = consulting_cost_usd + implementation_cost_usd
    roi_ratio           = (total_annual_saving - total_investment) / total_investment
    payback_months      = total_investment / (total_annual_saving / 12) if total_annual_saving > 0 else float("inf")

    return {
        "annual_cost_saving":    round(annual_cost_saving),
        "annual_failure_saving": round(failure_saving),
        "annual_hitl_saving":    round(hitl_saving),
        "total_annual_saving":   round(total_annual_saving),
        "total_investment":      round(total_investment),
        "roi_ratio":             round(roi_ratio * 100, 1),
        "payback_months":        round(payback_months, 1),
    }
```

## 6. Phase 4: Implementation — 구현 지원

### 6.1 구현 지원 방식

HAchillesWorld 컨설팅의 구현 지원은 세 가지 방식으로 제공된다.

```
방식 1: 가이드형 (Advisory)
  고객 팀이 직접 구현, 컨설턴트는 주 1회 리뷰 세션
  → 고객 역량이 충분하고, 비용을 최소화할 때

방식 2: 협업형 (Co-implementation)
  컨설턴트와 고객 엔지니어가 쌍을 이루어 공동 구현
  → 역량 이전을 극대화할 때

방식 3: 대행형 (Managed Implementation)
  컨설턴트가 주도, 고객은 검토·승인
  → 속도가 중요하거나, 초기 역량이 부족할 때
```

```
  ✅ 단위 테스트 (커버리지 목표 달성)
  ✅ 진단 점수 변화 측정 (전/후 비교)
  ✅ 스프린트 회고 문서 (1페이지)
```

### 6.3 구현 품질 기준 (Definition of Done)

모든 구현 태스크는 다음 기준을 충족해야 "완료"로 간주한다.

```
코드 품질:
□ 단위 테스트 통과 (신규 코드 커버리지 > 80%)
□ 타입 힌트 적용 (mypy strict 통과)
□ 린터 경고 없음 (ruff 통과)
□ 코드 리뷰 승인 (최소 1명)

기능 품질:
□ 목표 지표 달성 (수치로 측정 가능)
□ 실제 에이전트 로그로 검증 완료
□ 엣지 케이스 테스트 완료

운영 준비:
□ 하네스 규칙 업데이트 (영향받는 규칙 검토)
□ 모니터링 지표 추가 (새 컴포넌트 계측)
□ 롤백 계획 문서화
```

### 6.4 주요 구현 태스크 가이드

#### 태스크: EnsembleDynamics 구현 (L1→L2 핵심)

```python
# HAchillesWorld 책에서 제공된 EnsembleDynamics를 고객 도메인에 맞게 커스터마이즈

# 컨설턴트 가이드: 고객 도메인에 맞는 상태 차원 결정
# Digital Laws 도메인 예시: API 응답 상태
class DigitalWorldState:
    """디지털 API 환경의 World State."""
    api_response_time: float    # API 응답 시간 (ms)
    error_rate: float           # 현재 오류율
    queue_depth: int            # 대기 요청 수
    cache_hit_rate: float       # 캐시 히트율

# Physical Laws 도메인 예시: 로봇 창고 상태
class PhysicalWorldState:
    """물리 창고 환경의 World State."""
    robot_position: tuple       # (x, y, z)
    gripper_force: float        # 그립 강도 (N)
    obstacle_distances: list    # 주변 장애물 거리 목록
    battery_level: float        # 배터리 잔량

# 컨설턴트 지침:
# 1. 고객 도메인의 핵심 상태 변수를 3~7개로 압축한다
# 2. 각 변수는 연속형 또는 범주형으로 명확히 정의한다
# 3. World Model이 예측해야 하는 변수만 포함한다 (관측만 하는 변수 제외)
```

#### 태스크: HITL 트리거 기준 정의 (공통 핵심)

```python
# 고객 도메인에 맞는 HITL 트리거 기준 정의 워크시트

HITL_TRIGGERS = {

    "irreversible_action": {

        "threshold": None,
        "action": "require_human_approval",
        "urgency": "high",
    },
    
    # 비용 초과 트리거
    "budget_alert": {
        "condition": "daily_cost_usd > daily_budget * 0.80",
        "threshold": 0.80,
        "action": "notify_and_switch_model",
        "urgency": "normal",
    },
    
    # 도메인 특화 트리거 (고객과 공동 정의)
    # 예: 금융 도메인
    "large_transaction": {
        "condition": "transaction_amount > 10_000_000",  # 1천만원 이상
        "action": "require_dual_approval",

    },
}

# 컨설턴트 워크숍 질문:
# Q1: "에이전트가 절대 혼자 결정해서는 안 되는 상황은 무엇인가?"
#     → 항상 HITL 필요한 행동 목록 도출
# Q2: "에이전트가 잘못 결정했을 때 복구 비용이 가장 큰 경우는?"

# Q3: "현재 운영팀이 하루에 처리할 수 있는 HITL 요청은 최대 몇 건인가?"
```

## 9. 인게이지먼트 모델 및 가격

### 9.1 인게이지먼트 프로세스

```
1. 초기 문의 → 무료 퀵스캔 (S1)
   └── 가치 확인 → 구체적 인게이지먼트 제안

2. 제안서 (SOW, Statement of Work)
   ├── 범위 (Scope of Work)
   ├── 결과물 (Deliverables) 목록
   ├── 일정 (Timeline)
   ├── 가격 및 지불 조건
   └── 역할과 책임 (RACI)

3. 계약 체결
   ├── 비밀유지계약 (NDA)
   ├── 데이터 처리 계약 (DPA)
   └── 서비스 계약 (MSA + SOW)

4. 인게이지먼트 실행
   └── 주간 상태 업데이트, 월간 리뷰

5. 클로즈아웃
   ├── 최종 결과물 인도
   ├── 지식 이전 완료 확인
   └── 사후 만족도 조사 (30일, 90일)
```

### 9.2 가격 정책

#### 시간 기반 요금

| 역할 | 시간당 요금 | 일당 요금 |
|------|-----------|---------|
| 시니어 컨설턴트 | ₩300,000 | ₩2,400,000 |
| 리드 컨설턴트 | ₩500,000 | ₩4,000,000 |
| 파트너/전문가 | ₩800,000 | ₩6,400,000 |

#### 프로젝트 기반 요금 (표준)

| 서비스 | 기간 | 요금 | 포함 내용 |
|--------|------|------|-----------|
| S2: 표준 진단 | 1주 | ₩5,000,000 | 리포트 1부, 발표 1회 |
| S3: 심층 감사 | 4주 | ₩20,000,000 | 리포트 1부, 워크숍 2회 |

#### 성과 기반 요금 (선택 옵션)

고객 요청 시 기본 요금의 70% + 성과 달성 시 보너스 30% 구조를 제공한다.

```
성과 지표 예시:
  - 진단 점수 목표 달성 (예: 67 → 85+)
  - 비용 절감 목표 달성 (예: 30% 절감)
  - Drift Rate 목표 달성 (예: 30% → 5% 미만)

보너스 지급 조건:
  인게이지먼트 종료 후 90일 내 목표 달성 시 보너스 지급
```

### 9.3 계약서 핵심 조항

```
① 데이터 처리 조항
   - 고객 운영 로그는 진단 목적으로만 사용
   - 분석 완료 후 30일 이내 삭제 또는 반환
   - 고객 사례 공개 시 사전 서면 동의 필요

② 결과물 소유권
   - 컨설팅 중 생성된 고객 특화 코드: 고객 소유
   - HAchillesWorld 방법론 및 프레임워크: HAchillesWorld 소유
   - 공동 개발 IP: 별도 협의

③ 책임 제한
   - 자동 생성 하네스 규칙의 프로덕션 적용 전 고객 검토 의무
   - 컨설턴트 권고를 따르지 않아 발생한 손해는 고객 책임
   - 최대 책임 한도: 해당 인게이지먼트 총 계약금액

④ 비경쟁 조항
   - 컨설턴트는 인게이지먼트 기간 + 6개월간 동종 경쟁사 직접 지원 금지
```
