## 1. 소개

### HAchillesWorld란?

HAchillesWorld는 AI 에이전트 시스템의 **World Model 품질을 진단하고 최적화하는 플랫폼**이다.

에이전트가 작동하는 것을 확인하는 것은 쉽다.  
에이전트가 *올바른 이유로* 작동하는지 확인하는 것은 어렵다.

HAchillesWorld는 두 번째 질문에 답한다.

### 3가지 핵심 기능

```
┌─────────────────────────────────────────────────────────┐
│  Scan      지금 내 에이전트는 어느 수준인가?            │
│  ─────     Levels × Laws 좌표계로 10분 안에 진단       │
├─────────────────────────────────────────────────────────┤
│  Optimize  다음 단계로 가려면 무엇을 해야 하는가?       │
│  ────────  자동화된 로드맵 + 하네스 코드 생성           │
├─────────────────────────────────────────────────────────┤
│  Operate   지금 에이전트가 잘 작동하고 있는가?          │
│  ───────   실시간 드리프트 감지 + Replay 디버깅         │
└─────────────────────────────────────────────────────────┘
```

### Levels × Laws 프레임워크

HAchillesWorld의 모든 진단은 이 2차원 좌표계를 기준으로 한다.

```
         Physical  Digital  Social  Scientific
         Laws      Laws     Laws    Laws
         ─────────────────────────────────────
L1       단일 스텝 예측만 가능
Predictor

L2       멀티스텝 롤아웃, 반사실 추론 가능
Simulator

L3       자율 루프, 자기 수정 가능
Evolver
```

**내 에이전트는 어디에 있는가?** — Scan 모듈이 10분 안에 답한다.

## 4. Module 1: Scan — 진단

### 4.1 개요

Scan은 에이전트 로그와 설정을 분석해 15개 지표를 계산하고, Levels × Laws 좌표와 종합 점수를 담은 `DiagnosticReport`를 반환한다.

### 4.2 기본 사용법

```python
from hachillesworld import HAchillesWorldClient

client = HAchillesWorldClient()

# 로그와 설정으로 진단
report = client.scan(
    logs=log_data,          # list[dict] — 에이전트 이벤트 로그
    config=agent_config,    # dict      — 에이전트 설정
    agent_name="my-agent"   # str       — 에이전트 이름
)

# 결과 출력
print(report.summary())

# 상세 접근
print(f"Level: {report.level_label}")                  # 예: "L2.3"
print(f"Domain: {report.laws_domain}")                 # 예: "digital"
print(f"Score: {report.composite_score:.0f}/100")
print(f"WM Quality: {report.world_model_quality.score:.0f}")
print(f"Agency: {report.agency_level.score:.0f}")
print(f"Operations: {report.operational_health.score:.0f}")

# 즉시 조치 필요 항목
for issue in report.critical_issues:
    print(f"🔴 {issue.name}: {issue.description}")

# 권장 사항
for rec in report.recommendations:
    print(f"• {rec}")
```

### 4.3 진단 지표 15개 설명

#### Category A: World Model 품질

| 지표 | 의미 | 정상 | 경고 | 위험 |
|------|------|------|------|------|
| Prediction Error Rate | World Model 예측이 현실과 얼마나 다른가 | < 0.15 | 0.15~0.30 | > 0.30 |
| Calibration ECE | 확신도가 실제 정확도와 일치하는가 | < 0.10 | 0.10~0.20 | > 0.20 |
| Simulation Drift Rate | 재보정이 얼마나 자주 필요한가 | < 5% | 5~20% | > 20% |
| Uncertainty Coverage | 불확실성 구간이 실제 오차를 포함하는가 | > 80% | 60~80% | < 60% |
| Planning Depth | 내부 시뮬레이션을 몇 스텝 내다보는가 | ≥ 5스텝 | 2~5스텝 | < 2스텝 |

#### Category B: 에이전시 수준

| 지표 | 의미 | 정상 | 경고 | 위험 |
|------|------|------|------|------|
| Self-Correction Rate | 반성 단계에서 실제 수정을 적용하는가 | ≥ 10% | > 0% | 0% |
| Uncertainty Awareness | 계획 시 불확실성을 명시적으로 표현하는가 | 존재 | — | 미존재 |
| Goal Consistency | 목표가 일관되게 유지되는가 | > 90% | 75~90% | < 75% |
| Env Adaptation Speed | 환경 변화 후 얼마나 빨리 적응하는가 | ≤ 10스텝 | 10~30스텝 | > 30스텝 |
| Harness Coverage | 하네스 제약 규칙이 충분히 정의됐는가 | ≥ 20개 | 5~20개 | < 5개 |

#### Category C: 운영 건전성

| 지표 | 의미 | 정상 | 경고 | 위험 |
|------|------|------|------|------|
| Recalibration Frequency | 재보정이 과도하게 발생하는가 | < 5% | 5~20% | > 20% |
| Cost Efficiency | 예산 대비 비용 사용률 | ≤ 80% | 80~100% | > 100% |
| HITL Trigger Rate | 인간 개입이 과도하게 요청되는가 | < 5% | 5~20% | > 20% |
| Harness Violation Attempts | 금지 행동 시도가 발생하는가 | 0건/일 | 1~5건/일 | > 5건/일 |
| Checkpoint Recovery Rate | 장애 후 복구가 성공적인가 | > 98% | 90~98% | < 90% |

### 4.4 DiagnosticReport 구조

```python
@dataclass
class DiagnosticReport:
    agent_name:           str           # 에이전트 이름
    level:                Level         # L1 / L2 / L3
    level_progress:       float         # 0.0~1.0 (해당 Level 내 진행도)
    laws_domain:          LawsDomain    # physical/digital/social/scientific
    
    world_model_quality:  CategoryScore # WM 품질 카테고리
    agency_level:         CategoryScore # 에이전시 수준 카테고리
    operational_health:   CategoryScore # 운영 건전성 카테고리
    
    recommendations:      list[str]     # 권장 사항 목록
    metadata:             dict          # 추가 메타데이터
    
    # 프로퍼티 (자동 계산)
    composite_score:      float         # 종합 점수 (0~100)
    level_label:          str           # 예: "L2.3"
    critical_issues:      list[MetricScore]  # 즉시 조치 필요 항목
```

### 4.5 결과 해석 가이드

#### 점수별 해석

| 점수 | 상태 | 의미 |
|------|------|------|
| 80~100 | 🟢 양호 | 현재 Level에서 안정적. 다음 Level 준비 가능 |
| 60~79  | 🟡 주의 | 개선 필요 항목 존재. 단기 로드맵 실행 권장 |
| 0~59   | 🔴 위험 | 즉시 조치 필요. 즉시 Optimize 모듈 활용 권장 |

#### Level별 의미

| Level | 의미 | 다음 단계 |
|-------|------|-----------|
| L1.x | 단일 스텝 예측만 가능 | 앙상블 + MCTS 구현 → L2 진입 |
| L2.x | 멀티스텝 시뮬레이션 가능 | 자기 수정 루프 → L3 진입 |
| L3.x | 자율 루프 작동 | 안전성 강화 + 범위 확대 |

## 6. Module 3: Operate — 운영

### 6.1 실시간 Drift 모니터링

```python
from hachillesworld.operate.monitor import DriftMonitor

# 모니터 생성
monitor = DriftMonitor(
    agent_name="production-agent",
    threshold=0.15,           # 이 값 초과 시 드리프트로 판단
    window_size=20,           # 최근 N개 기록으로 안정성 판단
    alert_rate_threshold=0.20 # 윈도우의 20% 이상 드리프트 시 경보
)

# 경보 핸들러 등록
def on_drift_alert(alert):
    print(f"⚡ Drift Alert!")
    print(f"   에이전트: {alert.agent_name}")
    print(f"   드리프트: {alert.drift_value:.4f}")
    print(f"   최근 비율: {alert.recent_rate:.2%}")
    print(f"   권장 조치: {alert.recommended_action}")
    # → Slack 알림, 자동 재보정 트리거, 로그 기록 등

monitor.on_alert = on_drift_alert

# DEOR 루프에서 매 스텝 호출
for step in range(n_steps):
    action          = agent.plan(state, goal)
    result          = agent.execute(action)
    
    predicted_state = world_model.predict(state, action)
    actual_state    = env.observe()
    
    # 드리프트 기록 (딕셔너리로 상태 전달)
    drift = monitor.record(
        predicted={"value": predicted_state.value, "cost": predicted_state.cost},
        actual={"value": actual_state.value, "cost": actual_state.cost},
    )
    
    # 드리프트가 크면 재보정
    if drift > monitor.threshold:
        world_model.recalibrate(actual_state)
    
    # 안정성 확인
    if not monitor.is_stable():
        print("⚠️ 에이전트 불안정 — 점검 필요")

# 운영 요약
summary = monitor.summary()
print(f"총 기록: {summary['total_records']}건")
print(f"경보 횟수: {summary['alert_count']}건")
print(f"최근 드리프트율: {summary['recent_drift_rate']:.2%}")
print(f"안정 상태: {'✅' if summary['is_stable'] else '⚠️'}")
```

### 6.2 Replay 디버깅

실패한 에피소드를 단계별로 재생해 근본 원인을 찾는다.

```python
from hachillesworld.operate.replay import ReplayDebugger
from hachillesworld.core.models import AgentEvent
import time

# 방법 1: AgentEvent 객체 목록
events = [
    AgentEvent("my-agent", "plan",    time.time(), {"planning_depth": 10, "uncertainty": 0.08}),
    AgentEvent("my-agent", "execute", time.time(), {"action": "update_inventory"}),
    AgentEvent("my-agent", "observe", time.time(), {"prediction_error": 0.42}),  # 이상
    AgentEvent("my-agent", "reflect", time.time(), {"recalibrated": False}),
]

# 방법 2: 딕셔너리 목록 (기존 로그 형식)
events = [
    {"agent_name": "my-agent", "event_type": "plan",
     "timestamp": 1748745600.0, "payload": {"uncertainty": 0.08}},
    # ...
]

# 세션 로드
debugger = ReplayDebugger()
session  = debugger.load(episode_id="ep-2026-001", events=events)

# 터미널 출력
debugger.print_session(session)

# 프로그래밍 방식으로 접근
print(f"근본 원인: {session.root_cause}")
print(f"실패 스텝: Step {session.failure_step}")
print(f"이상 프레임 수: {len(session.anomaly_frames)}")

for frame in session.anomaly_frames:
    print(f"Step {frame.step} [{frame.event_type}]: {frame.anomaly_reason}")
```

```
    Step   1 [execute   ]
       action: update_inventory

  🔴 Step   2 [observe   ]
       ⚠ Simulation Drift 0.4200 > 0.15 — 재보정 미실행 여부 확인
       prediction_error: 0.42

    Step   3 [reflect   ]
       recalibrated: False

  이상 프레임: 1건 / 실패 스텝: Step 2
```

### 6.3 Meta-Harness 자동 학습

반복 실패에서 새 하네스 규칙을 자동으로 학습한다.

```python
from hachillesworld.operate.meta_harness import MetaHarness

# Meta-Harness 초기화
meta = MetaHarness(
    auto_apply_threshold=10,    # 10회 이상 반복 실패 시 규칙 제안
    human_review_required=True  # 자동 적용 전 인간 검토 필요
)

# 실패 이벤트 기록 (에이전트 실패 시 호출)
def on_agent_failure(event: dict):
    meta.record_failure(event)

# 예시: 재고 API 지연 실패 패턴
for _ in range(12):
    meta.record_failure({
        "event_type": "observe:api_timeout",
        "payload": {
            "description": "inventory_api 응답 지연 > 3초",
            "affected_domain": "inventory",
        }
    })

# 대기 중인 규칙 확인
pending = meta.get_pending_rules()
for rule in pending:
    print(f"제안된 규칙: {rule.rule_id}")
    print(f"  조건: {rule.condition}")
    print(f"  대응: {rule.action}")
    print(f"  강도: {rule.severity}")
    print()

# 검토 후 승인 또는 거부
rule_id = pending[0].rule_id
meta.approve_rule(rule_id)   # 적용
# meta.reject_rule(rule_id)  # 거부

# 요약
summary = meta.summary()
print(f"감지된 패턴: {summary['total_patterns']}개")
print(f"대기 규칙: {summary['pending_rules']}개")
print(f"적용된 규칙: {summary['applied_rules']}개")
```

## 7. SDK 통합 가이드

### 7.1 @instrument 데코레이터 — 가장 간단한 통합

에이전트 클래스에 데코레이터 하나만 붙이면 자동으로 계측된다.

```python
from hachillesworld import HAchillesWorldClient, instrument

client = HAchillesWorldClient(api_key="haw-...")

@instrument(client, agent_name="supply-chain-agent")
class SupplyChainAgent:
    """
    기존 에이전트 코드를 그대로 유지한다.
    plan / execute / observe / reflect 메서드만 있으면 자동 계측.
    """
    
    def plan(self, state: dict, goal: str) -> str:
        # HAchillesWorld가 자동으로 기록:
        # - 호출 시각
        # - 소요 시간
        # - 오류 발생 여부
        return self._select_action(state, goal)
    
    def execute(self, action: str) -> dict:
        return self._run_action(action)
    
    def observe(self, result: dict) -> dict:
        return self._update_state(result)
    
    def reflect(self, observation: dict) -> None:
        self._update_world_model(observation)
    
    def _select_action(self, state, goal): ...
    def _run_action(self, action): ...
    def _update_state(self, result): ...
    def _update_world_model(self, observation): ...
```

> **주의**: `@instrument`는 `plan`, `execute`, `observe`, `reflect` 이름의 메서드만 자동 계측한다. 다른 메서드는 영향받지 않는다.

### 7.2 수동 이벤트 기록

계측 메서드 이름이 다르거나 세부 정보를 추가로 기록하려면 수동 방식을 사용한다.

```python
import time
from hachillesworld import HAchillesWorldClient
from hachillesworld.core.models import AgentEvent

client = HAchillesWorldClient()

# 방법 1: track 헬퍼 (간단)
client.track(
    agent_name="my-agent",
    event_type="plan",
    payload={
        "planning_depth": 15,
        "uncertainty": 0.12,
        "selected_action": "reorder_inventory",
    }
)

# 방법 2: emit 직접 호출 (상세)
client.emit(AgentEvent(
    agent_name="my-agent",
    event_type="observe",
    timestamp=time.time(),
    payload={
        "prediction_error": 0.08,
        "error_within_uncertainty": True,
        "actual_inventory": 820,
        "predicted_inventory": 847,
        "drift": 0.032,
    },
    trace_id="trace-abc123",
))

# 버퍼 강제 플러시 (프로그램 종료 전 또는 주기적으로 호출)
flushed_count = client.flush()
print(f"{flushed_count}개 이벤트 전송 완료")
```

## 9. 로그 형식 가이드

### 9.1 표준 이벤트 타입

HAchillesWorld는 다음 이벤트 타입을 인식한다.

| 이벤트 타입 | 의미 | 주요 payload 키 |
|-------------|------|----------------|
| `plan` | 에이전트가 다음 행동을 계획 | `planning_depth`, `uncertainty`, `confidence` |
| `execute` | 행동을 실제 환경에서 실행 | `action`, `cost_usd` |
| `observe` | 실행 결과를 관측 | `prediction_error`, `error_within_uncertainty`, `goal_achieved` |
| `reflect` | 관측 결과를 반성, WM 업데이트 | `recalibrated`, `correction_applied`, `goal_changed` |
| `hitl_request` | 인간 개입 요청 | `reason`, `urgency` |
| `harness_violation` | 하네스 규칙 위반 시도 | `attempted_action`, `rule_id` |
| `checkpoint_recovery` | 체크포인트에서 복구 시도 | `recovery_success`, `checkpoint_id` |
| `error` | 예외/오류 발생 | `error_type`, `error_message` |

### 9.2 표준 로그 레코드 형식

```json
{
  "agent_name": "my-agent",
  "event_type": "observe",
  "timestamp": 1748745600.123,
  "payload": {
    "prediction_error": 0.09,
    "error_within_uncertainty": true,
    "goal_achieved": false,
    "cost_usd": 0.0012,
    "trace_id": "trace-abc123"
  }
}
```

### 9.3 최소 필수 필드

진단 정확도를 위해 최소한 다음 필드가 포함되어야 한다.

```
필수:
  event_type (str)   — "plan" / "execute" / "observe" / "reflect"
  
강력 권장:
  payload.planning_depth  (plan 이벤트)   — L1/L2/L3 판정에 사용
  payload.prediction_error (observe 이벤트) — WM 품질 측정에 사용
  payload.uncertainty      (plan 이벤트)   — 에이전시 수준 측정에 사용
  payload.recalibrated     (reflect 이벤트) — Drift Rate 측정에 사용
```

### 9.4 기존 로그를 HAchillesWorld 형식으로 변환

```python
# 기존 로그 형식이 다를 경우 변환 함수 작성
def convert_my_logs(raw_logs: list[dict]) -> list[dict]:
    converted = []
    for log in raw_logs:
        converted.append({
            "event_type": log.get("type", "unknown"),
            "timestamp":  log.get("ts", 0.0),
            "payload": {
                "planning_depth":  log.get("depth", 1),
                "prediction_error": log.get("pred_err", 0.0),
                "uncertainty":      log.get("unc", 0.0),
                "recalibrated":     log.get("recal", False),
            }
        })
    return converted

# 변환 후 진단
logs   = convert_my_logs(raw_logs)
report = client.scan(logs=logs, config=config, agent_name="my-agent")
```

## 11. 자주 묻는 질문 (FAQ)

**Q: API 키 없이 사용할 수 있나요?**

A: 네. 로컬 진단(Scan), 로드맵 생성(Optimize), Drift 모니터링(Operate)은 모두 API 키 없이 동작합니다. API 키는 클라우드 이벤트 수집 및 팀 공유 기능에만 필요합니다.

---

**Q: 로그가 없는데 진단할 수 있나요?**

A: 빈 로그로도 진단이 가능하지만, 정확도가 낮습니다. 최소 20개 이상의 이벤트 로그가 있어야 의미 있는 진단 결과를 얻을 수 있습니다. 먼저 `@instrument` 데코레이터로 로그를 수집한 후 진단하는 것을 권장합니다.

---

**Q: L2 판정을 받았는데 실제로는 L1 같아요. 왜 그런가요?**

A: Level 판정은 로그에서 측정 가능한 지표를 기반으로 합니다. `planning_depth` 값이 로그에 없으면 기본값(1)이 사용돼 L1으로 낮게 판정될 수 있습니다. 로그 형식에 `planning_depth` 키를 추가해주세요.

---

**Q: DriftMonitor의 threshold 값을 어떻게 설정해야 하나요?**

A: 처음에는 기본값(0.15)으로 시작하고, 일주일간 운영 후 평균 drift 값을 기준으로 조정하세요. 일반적으로 `평균 drift × 2`를 임계값으로 설정하면 적절합니다. 너무 낮으면 과잉 경보, 너무 높으면 실제 드리프트를 놓칩니다.

---

**Q: 생성된 하네스 규칙을 바로 프로덕션에 적용해도 되나요?**

A: **아니오.** 자동 생성된 규칙은 초안입니다. 반드시 담당 엔지니어가 검토한 후 적용해야 합니다. 특히 `severity="hard"` 규칙은 에이전트를 완전히 차단할 수 있으므로 신중하게 검토하세요.

---

**Q: 여러 에이전트를 동시에 진단할 수 있나요?**

A: 네. 에이전트별로 별도의 `client.scan()` 호출을 하거나, 에이전트별로 로그를 분리해 각각 진단하면 됩니다.

```python
for agent_name, agent_logs in all_agents_logs.items():
    report = client.scan(logs=agent_logs, config=configs[agent_name], agent_name=agent_name)
    print(report.summary())
```

---

**Q: Python 3.11 미만에서도 동작하나요?**

A: 지원하지 않습니다. `match/case`, `list[dict]` 타입 힌트 등 Python 3.10+ 문법을 사용합니다. Python 3.11 이상을 권장합니다.
