## 1부: 전문가별 1차 독립 분석

### [김지훈 — World Model / AI 아키텍처 전문가]

**총평**: 개념적으로 흥미롭지만, 기술적 엄밀성에서 중대한 허점이 있다.

#### 강점
- "World Model"이라는 개념을 에이전트 진단 도구로 연결한 프레이밍 자체는 독창적이다. 학계에서 논의되는 개념을 실무 진단 도구로 번역한 시도는 가치 있다.
- Planning Depth + MCTS 조합은 실제로 AlphaGo, MuZero 계열에서 검증된 접근이다. 이를 LLM 에이전트에 적용하려는 방향은 맞다.
- Calibration ECE (Expected Calibration Error)는 통계적으로 잘 정의된 지표다. LLM의 과신(overconfidence) 문제에 적용하는 것은 합리적이다.

#### 문제점 (심각도 순)

**[P1 — 치명적] "World Model"의 정의가 흐리다**

Yann LeCun(Meta)의 World Model 정의는 매우 구체적이다: JEPA(Joint Embedding Predictive Architecture) 기반의 자기지도 학습 표현 모델. DeepMind의 Dreamer 시리즈도 World Model을 latent-space dynamics model로 엄밀하게 정의한다.

HAchillesWorld가 말하는 "World Model"은 이와 다르다. 에이전트의 "세상에 대한 이해"를 포괄적으로 지칭하는 **메타포**로 사용하고 있다. 이는 학계 용어와 충돌하며, 기술적 전문가가 검토할 때 즉각적인 신뢰 하락을 일으킨다.

> 예: 컴퓨터 비전 연구자가 보면 "이 사람들이 World Model을 제대로 이해하는가?" 라는 의문을 품는다.

**[P2 — 심각] 15개 지표의 도출 근거가 없다**

15개 지표가 어떻게 선정됐는지, 왜 15개인지, 각 지표 간 독립성이 어떻게 보장되는지 어디에도 설명이 없다. 특히:

- **Simulation Drift**: 무엇 대비의 drift인가? Ground truth는 무엇인가? 어떻게 측정하는가?
- **Planning Depth**: MCTS를 LLM 에이전트에 적용할 때 state space는 어떻게 정의하는가? 이 부분이 가장 기술적으로 어렵다.
- **Harness Coverage**: "알려진 실수"의 전체 집합을 어떻게 정의하는가? 분모가 없는 비율은 의미없다.

**[P3 — 보통] L1/L2/L3 레벨 분류의 과학적 근거 미흡**

"L2.3 × Digital Laws" 같은 정밀한 소수점 판정이 어디서 나오는가? 해당 판정 기준이 공개되지 않으면 블랙박스 평가다. 기업이 "우리 에이전트가 L2.3이다"라는 결과를 받았을 때 재현 가능한가?

**[P4 — 보통] MCTS와 LLM 에이전트 통합의 실제 구현 난이도**

MCTS는 잘 정의된 state space와 action space, 그리고 빠른 rollout simulator가 필요하다. LLM 에이전트에서 이를 구현하는 것은 매우 비자명하다. `mcts_planner.py`를 자동 생성한다고 주장하지만, 실제 기업 환경에서 plug-and-play로 동작할 가능성은 낮다.

#### 권고사항
1. "World Model"을 정의하는 명확한 technical paper 또는 whitepaper 게시 필요
2. 15개 지표의 측정 방법론과 도출 근거를 공개적으로 문서화
3. 최소 2개 이상의 실제 에이전트에서의 사례 검증(ablation study 수준)
4. LeCun/Dreamer와의 개념적 차별점 명시

### [이수진 — UX·웹 디자인 전문가]

**총평**: 기술적 감성은 뛰어나지만, B2B 전환율을 최우선으로 설계된 사이트는 아니다.

#### 강점
- 다크 테마 + 보라/청록 그라디언트 조합: AI/Tech 감성이 잘 살아있다.
- 터미널 모의 스캔 애니메이션: 개발자 타겟에게 즉각적인 공감대 형성.
- 블로그 글 품질이 높다. 특히 MCTS 글은 일반인도 이해할 수 있는 비유가 훌륭하다.
- TOC 사이드바, 스크롤 페이드인 등 세부 UX 완성도 높음.

#### 문제점

**[P1 — 치명적] Scan 폼이 진짜 백엔드와 연결되지 않는다**

사용자가 폼을 작성하고 제출하면 미리 생성된 더미 리포트가 나온다. 에이전트 이름만 치환되고 나머지 내용은 동일하다. 이를 B2B 의사결정권자가 발견하는 순간 신뢰가 완전히 무너진다.

```
현재 플로우: 폼 입력 → sessionStorage 저장 → 고정된 리포트 페이지
문제: 어떤 에이전트 이름을 넣어도 점수는 항상 74/100, 동일한 결과
```

이것은 "MVP 데모"라고 투명하게 밝혀야 한다. 그렇지 않으면 기만이다.

**[P2 — 심각] 모바일 경험이 사실상 없다**

`.nav-links { display:none }` — 모바일에서 네비게이션이 완전히 사라진다. 햄버거 메뉴도 없다. 2026년 기준으로 모바일 B2B 탐색 비율은 40%를 넘는다. 링크드인에서 블로그 글을 본 사람이 모바일로 접속하면 즉각 이탈한다.

**[P3 — 심각] 사회적 증거(Social Proof)가 사실상 없다**

수치는 많다 (67→91, ₩48M, 3.8% drift). 그런데 이를 증명할 고객사 이름, 담당자 인용, 로고가 하나도 없다. 익명 처리하더라도 "대형 전자상거래 기업 A사 담당자" 같은 최소한의 맥락이 필요하다.

**[P4 — 보통] Operate 대시보드 링크가 Nav에 노출돼 혼란**

Operate는 Scan 후 자연스럽게 진입해야 하는 페이지인데, 상단 내비에 직접 링크가 걸려있다. operate.html에 직접 접근하면 세션 데이터가 없어 더미 데이터가 표시된다. 첫 방문자에게 제품의 실체가 노출된다.

**[P5 — 보통] 접근성(Accessibility) 고려 미흡**

- 색상 대비: `--muted: #94a3b8` 텍스트가 배경 대비 WCAG AA 기준 미달 가능성
- `<span class="nav-logo">` — 링크가 아닌 span 사용. 키보드 탐색 불가
- `alt` 없는 이미지들, ARIA 레이블 미사용

#### 권고사항
1. 폼이 실제 백엔드로 연결될 때까지 "미리보기 모드 / 샘플 리포트 보기"로 명확히 표기
2. 모바일 햄버거 메뉴 즉시 구현
3. Operate 직접 접근 차단 또는 "샘플 데이터로 보기" 배너 추가
4. 고객 사례 위젯 최소 1개 (익명 가능) 히어로 섹션 아래 배치
5. WCAG AA 접근성 감사

### [박도현 — B2B 마케팅 전문가]

**총평**: 콘텐츠 전략은 좋다. 그런데 누구에게 팔 것인지가 흐릿하다.

#### 강점
- 블로그 글 3편의 방향이 좋다: 증상(실패) → 기술(MCTS) → 전략(도입)으로 자연스러운 퍼널 형성.
- "진단 없는 최적화는 도박이다" 카피라이팅 강렬하다. 공감을 유발한다.
- CLI 터미널 스타일 데모는 개발자 커뮤니티에서 바이럴 가능성이 있다.
- "무료 진단" → 리포트 → 유료 전환의 PLG 구조는 방향성이 맞다.

#### 문제점

**[P1 — 치명적] ICP(Ideal Customer Profile)가 정의되지 않았다**

현재 사이트는 세 집단에게 동시에 말하고 있다:

| 타겟 후보 | 메시지 적합도 | 실제 구매 권한 |
|-----------|--------------|---------------|
| AI 개발자 (개인) | 높음 (CLI 스타일) | 낮음 (구매자가 아님) |
| MLOps 팀장 | 중간 | 중간 |
| CTO / AI 전략 임원 | 낮음 (너무 기술적) | 높음 |
| AI 컨설팅 회사 | 중간 | 높음 |

B2B SaaS에서 개발자에게 공감받는 것과 CFO가 예산을 승인하는 것은 다른 문제다. 지금은 개발자 공감은 되지만 CFO 설득 레이어가 없다.

**[P2 — 심각] 경쟁사와의 차별화가 명시되지 않았다**

다음 도구들과 HAchillesWorld는 어떻게 다른가? 사이트 어디에도 답이 없다:

- **Arize AI**: AI 에이전트 관찰 가능성(Observability) 플랫폼
- **LangSmith (LangChain)**: LLM 에이전트 트레이싱·평가
- **WhyLabs**: ML 모델 모니터링
- **Weights & Biases**: 실험 추적 + 에이전트 평가
- **Helicone**: LLM 비용·성능 모니터링

"World Model 품질을 측정한다"는 차별점이 있지만, 이게 왜 위 도구들보다 더 필요한지를 설명해야 한다.

**[P3 — 심각] "₩48M 절감" 수치의 출처와 재현 가능성**

가장 강력한 마케팅 수치인데, 맥락이 없다. 누구의 케이스인가? 어떻게 계산했는가? 이 수치 없이 들어온 사람에게 설득력이 있는가? 자세한 케이스 스터디 페이지가 없으면 신뢰보다 의심을 유발한다.

**[P4 — 보통] 한국어 우선 vs 글로벌 야심의 충돌**

책 제목은 영어("Agentic World Modeling 2027"), 비교 지표는 달러($18,000/년)와 원화(₩48M)가 혼재한다. 국내 시장에서 증명하고 글로벌로 나갈 것인지, 처음부터 글로벌인지 전략적 포지셔닝이 없다.

**[P5 — 보통] SEO 구조 취약**

- 모든 페이지 메타 description이 한글로만 되어 있다
- OG 이미지(Open Graph)가 없다 — 링크드인/카카오 공유 시 썸네일 없음
- 구조화 데이터(Schema.org) 미적용 — 검색엔진 rich snippet 불가
- 블로그 URL이 `/blog-mcts-planning-depth.html` 형태 — SEO 비친화적

#### 권고사항
1. ICP를 2개로 좁히기: ① MLOps 팀장 ② AI 컨설팅 회사 — 각각 다른 랜딩
2. "vs 경쟁사" 비교 섹션 또는 별도 페이지 추가
3. 케이스 스터디를 단독 페이지로 분리, 구체적 수치 + 방법론 공개
4. OG 이미지 생성 (블로그 글마다)
5. 한국어 우선 → 영어 버전 추가 로드맵 공개

### [최현아 — AI 도입 실무자 (제조·물류)]

**총평**: 문제 정의는 공감되지만, 현장에서 쓰기까지 넘어야 할 장벽이 크다.

#### 강점
- "에이전트 실패의 진짜 원인"이라는 문제 프레이밍은 현장에서 100% 공감한다. AI를 도입했는데 왜 실패했는지 모르는 경우가 정말 많다.
- "하네스(Harness)"라는 개념 — 알려진 실수에 대한 명시적 규칙 — 은 현장에서 즉시 유용하다.
- 3단계 Phase 로드맵 (진단 → 기반 구축 → 자율화)은 기업의 예산 승인 사이클에 맞다.

#### 문제점

**[P1 — 치명적] "에이전트 로그 파일"이 무엇인지 현장은 모른다**

`hachillesworld scan --logs ./logs.json` — 이 명령어에서 `logs.json`이 무엇인지, 어떤 형태여야 하는지, 어떻게 만드는지 설명이 전혀 없다.

현장에서 "AI 에이전트 로그"를 JSON으로 추출하는 것 자체가 이미 개발팀 작업이다. LangChain 에이전트, AutoGPT, 자체 개발 에이전트 — 각각 로그 포맷이 다르다. 파서가 있는가? 없다면 사실상 사용 불가.

**[P2 — 심각] 데이터 보안 및 개인정보 처리 방침 없음**

기업 에이전트 로그에는 고객 데이터, 내부 의사결정 데이터, 영업 비밀이 포함될 수 있다. HAchillesWorld에 로그를 업로드한다는 것은 매우 민감한 행위다.

- 데이터가 어디에 저장되는가?
- 얼마나 보관되는가?
- 제3자에게 공유되는가?
- GDPR, 개인정보보호법 준수 여부는?

이 정보가 없으면 어떤 대기업도 실제 데이터를 올리지 않는다.

**[P3 — 심각] 업종별 컨텍스트 없음**

제조업, 물류, 금융, 의료 — 각 업종마다 "World Model 품질"의 의미와 허용 오차가 다르다. 의료 AI는 ECE 0.05도 위험할 수 있고, 물류 AI는 Drift 10%가 허용될 수 있다. 동일한 15개 지표를 모든 업종에 동일하게 적용하는 것이 타당한가?

**[P4 — 보통] 24주 로드맵이 현실적인가?**

내 경험상 대기업에서 AI 에이전트 관련 변경 사항 하나를 배포하는 데 6~8주가 걸린다(검토·승인·테스트·배포). 24주에 L2→L3 달성이라는 주장은 스타트업이나 사내 독립 팀에서나 가능한 이야기다.

**[P5 — 보통] 현장 구현 지원 없음**

`harness_rules.py`, `mcts_planner.py`를 자동 생성해준다고 하는데, 이를 기존 에이전트 코드에 통합하는 작업은 누가 하는가? 컨설팅 서비스가 있는가? 없다면 도구를 받아도 현장에서 사장된다.

#### 권고사항
1. 로그 수집기(Log Collector) SDK를 최우선으로 제공 — LangChain, AutoGPT, 자체 개발 에이전트 지원
2. 개인정보처리방침 + 데이터 보안 정책 페이지 즉시 생성
3. 업종별 벤치마크 기준값 공개 (제조 / 물류 / 금융 / 고객응대)
4. 구현 지원 서비스(Implementation Support) 유료 옵션 추가
5. 로컬 실행(on-premise) 옵션 명시

### [정승규 — 지적재산권 변호사]

**총평**: 여러 법적 위험 요소가 있다. 당장 조치해야 하는 것과 중장기 이슈를 구분한다.

#### [즉시 조치 필요]

**[R1 — 매우 위험] "World Model" 상표 선점 여부 확인**

"World Model"은 Yann LeCun(Meta)이 수년간 공개적으로 사용해온 개념어다. Meta가 "World Model" 관련 상표를 출원했는지 확인해야 한다. 미국 USPTO, 한국 특허청 모두 검색 필요.

단, "HAchilles"와 결합된 "HAchillesWorld" 자체는 상표 등록 가능성이 있다. 즉시 출원 권고.

> 확인 필요: 한국 특허청 (KIPRIS), USPTO에서 "World Model", "HAchillesWorld" 검색

**[R2 — 위험] "Achilles" 브랜드와의 충돌**

"HAchilles"는 "Achilles"를 포함한다. Achilles는 타이어(Achilles Tire), 스포츠 용품 등 다수 기업이 상표를 보유하고 있다. "HA"를 앞에 붙여도 유사 상표 판단 여지가 있다. 전문가 검토 필요.

**[R3 — 위험] 성과 수치 광고 규정 준수**

"+34% 정확도", "-48% LLM 비용", "₩48M 절감" 등 구체적 수치를 광고에 사용하는 것은 한국 표시광고법, 미국 FTC 규정상 **실증 가능한 근거**가 있어야 한다. 근거 없는 수치 광고는 과장 광고로 제재받을 수 있다.

특히 "McKinsey 보고서에 따르면 68%가 기대 이하 성과" — 해당 보고서의 실제 인용이 정확한지, 맥락이 왜곡되지 않았는지 확인 필요.

**[R4 — 위험] 인증 프로그램의 법적 지위**

"HAchilles Certified AI Native (HCAN)" — 이것이 공인 자격증인지, 민간 인증인지 명확히 해야 한다. 한국에서 자격증을 발급하려면 국가기술자격법 또는 민간자격관리법에 따른 등록이 필요할 수 있다. 등록 없이 "인증" "자격증" 명칭을 사용하면 법적 문제가 생긴다.

최소한 "이 인증은 HAchillesWorld가 자체적으로 발급하는 민간 수료 인증이며, 국가 공인 자격증이 아닙니다"라는 고지가 필요하다.

#### [중장기 이슈]

**[R5 — 중간] "Levels × Laws" 프레임워크 보호**

이 프레임워크가 충분히 독창적이라면 저작권 또는 영업 비밀로 보호 가능하다. 공개 블로그에 세부 내용을 모두 공개하면 보호 범위가 줄어든다. 핵심 알고리즘 부분은 비공개로 유지하거나 특허 출원 검토.

**[R6 — 중간] 오픈소스 라이선스 관계**

`mcts_planner.py`, `harness_rules.py` 등 자동 생성 코드의 라이선스 정책이 없다. 사용자가 이 코드를 상업적으로 사용할 때 어떤 조건이 적용되는가? 명시적 라이선스 필요.

#### 권고사항
1. "HAchillesWorld" 상표 즉시 출원 (한국·미국)
2. 모든 수치 광고에 근거 자료 문서화 및 법무 검토
3. 인증 프로그램에 "민간 자격" 명시 고지 추가
4. 민간자격관리법 검토 후 필요시 등록
5. 생성 코드 라이선스 정책 문서화

### [한민서 — 경영 컨설팅 전문가]

**총평**: 비즈니스 모델이 아직 완성되지 않은 컨셉 단계다. 전략적 명확성이 필요하다.

#### 강점
- 시장 타이밍이 좋다. AI 에이전트 실패 사례가 급증하면서 진단·평가 도구에 대한 수요가 실제로 존재한다.
- PLG + 콘텐츠 마케팅 조합은 SaaS 스타트업에게 가장 효율적인 GTM이다.
- 저자가 직접 쓴 책("Agentic World Modeling 2027")은 강력한 신뢰 자산이 될 수 있다.

#### 문제점

**[B1 — 치명적] 실제로 작동하는 제품이 없다**

Scan → Report 흐름이 데모다. Optimize는 "얼리 액세스"다. Operate는 더미 데이터다. 인증 프로그램은 버튼만 있다. 즉, 지금 이 웹사이트는 **제품 없는 마케팅 사이트**다.

이것 자체가 나쁜 것은 아니다. "Fake it till you make it"은 린 스타트업의 전통적 방법론이다. 그러나 투명성 없이 실제 서비스처럼 보이는 것은 리스크다. 고객, 투자자, 미디어가 실사(due diligence)를 할 경우 문제가 된다.

**[B2 — 심각] 비즈니스 모델이 모호하다**

현재 제품이 만들어질 경우 수익 모델은 무엇인가?

| 가능한 모델 | 현재 신호 | 문제 |
|------------|----------|------|
| SaaS 구독 | 가격 페이지 없음 | BM 불명확 |
| 프로젝트 컨설팅 | 컨설팅 메뉴 없음 | 서비스 방식 미정 |
| 인증 수수료 | HCP ₩100K | 볼륨이 있어야 수익 |
| 도구 라이선스 | GitHub 오픈소스? | 수익 경로 불명확 |
| 책 판매 | 아직 출판 전 | 부수적 수익원 |

**[B3 — 심각] 경쟁 환경 분석 부재**

2026년 현재 AI 에이전트 모니터링 공간은 빠르게 포화되고 있다:

- **Arize AI** (시리즈 B, $38M): LLM 관찰 가능성
- **LangSmith**: LangChain 생태계 내 표준
- **Honeyhive**: 에이전트 평가·최적화
- **Braintrust**: LLM 평가 프레임워크
- **Weights & Biases**: 광범위한 ML 모니터링

HAchillesWorld가 이들과 어떻게 다른지, 왜 기업이 기존 도구 대신 HAchillesWorld를 선택해야 하는지의 논리가 필요하다. "World Model 전문성"만으로는 약하다.

**[B4 — 보통] CAC vs LTV 구조가 성립하는가?**

인증 1건 ₩100K, 컨설팅 수익 불명확. 무료 Scan 전환율이 1~3% (PLG 일반)라고 할 때, 이 모델로 지속 가능한 사업이 되려면 얼마나 많은 트래픽이 필요한가? 역산하면 현실성이 보인다.

**[B5 — 보통] 팀이 보이지 않는다**

박성훈이라는 개인 저자가 전면에 나오지만, 회사 조직, 팀 구성, 기술 역량이 어디에도 표시되지 않는다. B2B 엔터프라이즈 계약에서 "팀이 없는 1인 창업자"에게 수천만 원 프로젝트를 맡기는 것은 매우 어렵다.

#### 권고사항
1. "프리뷰/데모" 상태를 투명하게 고지 + 실제 제품 개발 로드맵 공개
2. 비즈니스 모델 의사결정: SaaS vs 컨설팅 vs 하이브리드 중 택일
3. 팀 페이지 또는 어드바이저 보드 추가 (신뢰 구축)
4. 3~5개의 실제 파일럿 고객 확보 후 케이스 스터디 제작
5. 투자 유치 또는 부트스트랩 전략 결정 후 자원 배분

### 라운드 1 — "World Model이라는 브랜드, 자산인가 부채인가"

**한민서**: 저는 마케팅 관점에서 "World Model"이라는 키워드가 현재 AI 업계에서 가장 뜨거운 단어 중 하나라고 봅니다. LeCun이 연일 이야기하고 있고, 일반 기업 AI 담당자들도 들어본 용어죠. 선점 효과가 있습니다.

**김지훈**: 동의하기 어렵습니다. 기술 전문가가 보면 오히려 역효과입니다. "World Model을 안다고 자처하는데 LeCun이 말하는 JEPA 아키텍처랑 다른 이야기를 하고 있다"라는 인식이 생깁니다. 기술 신뢰도가 핵심인 B2B 도구에서 이는 치명적일 수 있습니다.

**박도현**: 절충안이 있습니다. 의사결정권자(CTO, AI 임원)에게는 "World Model"이 훌륭한 앵커 단어입니다. 개발자 커뮤니티에는 기술적 엄밀성을 별도로 제공하는 whitepaper가 필요합니다. 타겟별로 메시지를 다르게 가져가야죠.

**정승규**: 상표 관점에서 "World Model"이라는 단어 자체는 보호하기 어렵습니다. "HAchillesWorld"라는 브랜드를 강하게 구축하되, "World Model"은 설명적 용어로 사용하는 것이 안전합니다.

**→ 합의**: "World Model"을 브랜드 이름으로 쓰지 않고, 설명 개념으로 유지. "HAchillesWorld"를 상표 등록하고 브랜드 정체성의 핵심으로 삼는다.

### 라운드 2 — "지금 당장 가장 위험한 것은 무엇인가"

**이수진**: 저는 Scan 폼이 가짜라는 것을 숨기는 것입니다. 한 명의 기업 의사결정권자가 이걸 발견하고 SNS에 올리면 신뢰가 한순간에 무너집니다.

**정승규**: 저는 인증 프로그램이 더 위험합니다. ₩100K~₩500K를 받고 민간 자격 미등록 상태로 "인증"을 발급하면 소비자 보호법 위반 소지가 있습니다. 한 명의 불만 고객이 신고하면 사업 자체가 멈출 수 있습니다.

**한민서**: 둘 다 맞는데, 저는 팀 부재를 가장 위험하게 봅니다. 투자자를 만나거나 기업 계약을 논의할 때 "1인 개인"이라는 것이 드러나면 실행력에 대한 의심이 생깁니다.

**최현아**: 현장 관점에서는 데이터 보안 정책 부재가 가장 위험합니다. 기업 고객이 실제로 에이전트 로그를 올리는 순간, 그 데이터 처리에 대한 법적 책임 문제가 즉시 발생합니다.

**→ 우선순위 합의**:
1. **즉시(1주일 이내)**: Scan 데모 상태 투명 고지 + 개인정보처리방침 + 인증 법적 고지
2. **단기(1개월)**: 모바일 메뉴, 데이터 보안 정책, 상표 출원
3. **중기(3개월)**: 실제 백엔드 개발, 파일럿 고객, 팀 구성

### 라운드 3 — "이 제품의 진짜 강점은 무엇이고 어떻게 활용해야 하는가"

**한민서**: 솔직히 말하면, 지금 가장 강력한 자산은 제품이 아니라 **박성훈이라는 저자**입니다. 《Agentic World Modeling 2027》이 출판되면 그게 최고의 마케팅 자산입니다.

**박도현**: 맞습니다. 책 → 블로그 → 강연 → 컨설팅 → SaaS 도구로 이어지는 "전문가 브랜드" 모델이 이 시점에서 가장 현실적입니다. 지금 당장 SaaS 경쟁을 하려면 자원이 너무 많이 든다.

**김지훈**: 그렇다면 기술 whitepaper 하나가 블로그 글 열 개보다 가치있습니다. "Levels × Laws 프레임워크의 수학적 기반"을 arXiv에 올리면 학계와 실무자 양쪽에서 신뢰를 얻을 수 있습니다.

**최현아**: 저는 컨설팅 서비스를 먼저 팔기를 권합니다. 실제 기업 프로젝트를 3~5건 수행하면서 도구를 만들어야 합니다. 도구가 먼저, 고객이 나중이 아니라 고객이 먼저, 도구가 나중입니다.

**이수진**: 웹사이트는 지금 컨설팅 신청 폼을 전면에 배치해야 합니다. "무료 진단 시작"이 아니라 "30분 무료 상담 예약"이 지금 단계에서 더 정직하고 효과적인 CTA입니다.

**정승규**: 컨설팅 계약서, NDA, 데이터 처리 계약서 템플릿을 먼저 준비하세요. 첫 고객이 생기는 순간 법적 준비가 없으면 계약을 진행할 수 없습니다.

**→ 합의된 전략 방향**: "SaaS를 꿈꾸지만, 당장은 전문가 브랜드 기반 컨설팅으로 시작한다"
